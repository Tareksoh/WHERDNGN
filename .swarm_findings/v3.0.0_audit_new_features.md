# v3.0.0 New-Feature Quality Audit

Scope: 12 releases of new surface area added since v1.6.1 (Easter, MSG_HEARTBEAT,
mid-round drop→bot fill, welcome + new slash commands, tooltip layer, banner fade,
BALOOT pulse, K.SAUDI_NAMES + K.ARABIC_FONT, persistent stats, Stats/History buttons,
lobby kick, Settings panel, AKA chat hint, drop-recovery, overcall stagger, "Thinking…"
indicator, per-rung escalation delays, bot variable pacing, `setLobbyTooltip`).

## TL;DR

- **Total findings: 21** (3 HIGH, 9 MEDIUM, 9 LOW)
- **Severity breakdown**: HIGH=3 / MEDIUM=9 / LOW=9
- **Top 3 most impactful**:
  1. **HIGH** — `K.HEARTBEAT` watchdog half-built. Host sends, peers timestamp, but
     `IsHostLikelyGone()` is read by **nothing** in the UI. The "host gone" warning
     banner promised in CHANGELOG was never wired. Solo-host crash still leaves the
     other 3 players staring at a frozen UI with no on-screen signal.
     *(Net.lua:3231 → no UI consumer)*
  2. **HIGH** — Arabic-font probe is wrong. `pcall(probe.SetFont, ...)` discards
     SetFont's return value and only checks `ok` (no error). SetFont silently
     returns false on missing files in WoW; pcall returns ok=true regardless.
     `_arabicAvailable` is therefore stuck at `true` whenever the user hasn't
     installed the (unbundled) Arabic font, so `B.UI.SaudiName(key)` confidently
     returns `"حكم Hokm"` Arabic forms that are then rendered with `K.CARD_FONT`
     (Arial Narrow), which lacks Arabic glyphs — exact failure mode of the v2.0.2
     hotfix, surviving on the FontString-level path. *(UI.lua:432-447)*
  3. **HIGH** — AKA partner-signal chat hint never fires for the host's own bot
     partner. `_OnAKA` is the only place the hint prints, and `fromSelf(sender)`
     short-circuits at line 4005 before reaching the print. So a solo host (the
     dominant scenario) never sees the BF-30 hint when their bot teammate AKAs —
     the very player it was designed to help. *(Net.lua:4005 vs 4067)*

## Per-feature verdict

### Easter.lua (v1.5.1)

- Tooltip / spec accuracy: ✓ (clear comments on disable/remove path)
- Edge cases:
  - Hook-install on 0.5s timer: if Net.lua loads later than 0.5s after Easter.lua,
    hook never installs and easter silently never works. Comment claims this is
    addressed; no retry safety-net. **LOW** *(Easter.lua:150-154)*
  - Per-skip 10% means 4 skip windows × 8 rounds × 0.10 ≈ 96.6% trigger rate per
    full game on a target seat. Inside-joke probably intended, but worth noting.
  - Photo overlay (`_overlay`) is a singleton — re-fires within 5s reset texture
    on the existing frame, no leak.
- **No additional findings.**

### MSG_HEARTBEAT (v1.8.0)

- Wire path: ✓
- Watchdog state computed: ✓ (`_lastHeartbeatAt`, `IsHostLikelyGone`)
- **Finding (HIGH)** — UI never consumes heartbeat state; no banner; no chat
  warning. `IsHostLikelyGone` and `SecondsSinceHostHeartbeat` have **zero** UI
  callers (`Grep IsHostLikelyGone -- UI.lua` returns 0). The user-facing fix
  promised in CHANGELOG is unwired. *(Net.lua:3231)*
- **Finding (LOW)** — No payload validation on `_OnHeartbeat`. Constants.lua:293
  declares "Payload: empty" but `N._OnHeartbeat(sender)` accepts any payload — a
  malicious peer flooding `MSG_HEARTBEAT;<garbage>` would just get silently
  accepted. Cheap because it's just a single timestamp write, but worth a no-op
  payload-length guard. *(Net.lua:3212-3217)*
- **Finding (LOW)** — Heartbeat self-stops on `PHASE_LOBBY` and `PHASE_IDLE` only.
  Between rounds (PHASE_GAME_END → user clicks "New Game" → reset to LOBBY) the
  ticker self-stops and is rearmed at HostStartRound. Acceptable, but in a
  PHASE_SCORE that drags on >45s while waiting for the host to advance, peers may
  flag the host as gone (heartbeat continues during PHASE_SCORE — actually OK
  since `S.s.phase != IDLE/LOBBY`). **No bug, but intricate.**

### Mid-round drop → bot fill (v1.8.0 + v2.0.0 + v2.2.0)

- Lobby-phase drop: kick path ✓
- Mid-round drop: bot replace + AFK timer cancel ✓
- Escalation-window drop: `MaybeRunBot()` re-dispatch ✓
- Overcall-window drop: synthesize WAIVE ✓
- SWA-permission-window drop (responder): synthesize ACCEPT ✓
- **Finding (MEDIUM)** — `WHEREDNGN.lua:672-681` checks `swaRequest.caller ~= seat`,
  meaning the recovery handles the case where a *responder* (not the caller)
  drops. But if the *caller* drops mid-SWA (their seat is replaced by a bot),
  the SWA request still has the old caller seat, the 5s SWA timer eventually
  fires `HostResolveSWA`, which uses `req.encodedHand` — the bot now at that
  seat owns a different hand than the caller's snapshot. Subtle: the encoded
  hand was pinned in the request struct so SWA validates against the original
  hand snapshot, but the seat's CURRENT hostHand differs (bot inherits hostHand,
  but `req.encodedHand` was set at caller's submit time). Arguably the
  encodedHand pin saves it. Worth a test pass.
- **Finding (MEDIUM)** — `PHASE_PREEMPT` is not in the explicit recovery branch
  list (DOUBLE/TRIPLE/FOUR/GAHWA/OVERCALL/SWA). Falls through to the trailing
  `MaybeRunBot()` at line 685, which does dispatch the preempt branch. But the
  human's old `BelTimer(kind="preempt_pass")` is *not* canceled — eventually
  fires `_HostBelTimeout(kind="preempt_pass")` which broadcasts a redundant
  `MSG_PREEMPT_PASS`. By that time bot has already decided. Idempotent but
  noisy. *(WHEREDNGN.lua:645-681)*
- **Finding (LOW)** — Lobby-time kick has no "you were kicked" notification to
  the kicked client. They just see their seat go away on the next MSG_LOBBY,
  with no chat hint. Will appear as an addon bug to the user. *(WHEREDNGN.lua:606-607)*

### `K.SAUDI_NAMES` + `B.UI.SaudiName(key)` (v2.0.0)

- **Finding (HIGH)** — Probe at UI.lua:441-444 uses
  `local ok = pcall(probe.SetFont, probe, K.ARABIC_FONT, 12, "")` and sets
  `_arabicAvailable = (ok == true)`. This only verifies SetFont didn't *raise*,
  NOT that the font *loaded*. In WoW, calling `SetFont` with a missing path
  silently returns false without raising. So `_arabicAvailable = true`
  whenever the user hasn't manually downloaded `NotoNaskhArabic-Regular.ttf`
  (which is **not bundled** — see Constants.lua:46-49). SaudiName then
  returns `"حكم Hokm"` Arabic strings.
- **Finding (HIGH)** — Even if `_arabicAvailable` were correctly false, the
  buttons rendering SaudiName output (Hokm, Sun, etc.) use `K.CARD_FONT =
  "Fonts\\ARIALN.TTF"` (Arial Narrow, no Arabic glyphs) via `b:GetFontString()
  :SetFont(K.CARD_FONT, ...)` at UI.lua:403. So Arabic chars in the label
  string render as `□□□` boxes — the exact failure mode of v2.0.2.
  Combined with the broken probe above, the addon will likely show
  `□□□ Hokm` boxes on most installs. The proper fix needs the probe to
  capture and check SetFont's return value (`local ok, ret = pcall(...);
  _arabicAvailable = ok and (ret == true or ret == 1)`) AND the button-
  label rendering path needs to use a font that has the Arabic block, OR
  SaudiName must always return romanized.

### AKA partner-signal chat hint (v2.0.0 BF-30)

- Saudi rule correctness: ✓ (partner direction = "don't over-trump",
  opp direction = "they claim boss")
- **Finding (HIGH)** — On a solo-host game, a host bot calling AKA never
  prints the chat hint to the host's own client. `_OnAKA` is the only print
  site, and `fromSelf(sender)` blocks at line 4005 before reaching the print
  at 4067. The local human host only sees the hint when a NON-host bot
  calls AKA via the wire. In the dominant single-player solo-host path
  with bot partners, the BF-30 fix delivers nothing. *(Net.lua:4005, 4067)*
- **Finding (MEDIUM)** — The chat hint also fires on the *replay* path
  (resync). When a /reload triggers MSG_AKA replay frames from the host,
  every replayed AKA prints "X (partner) called AKA Y" to the user's chat,
  flooding them with stale messages they already saw before the reload.
  The `isReplay` flag is captured at line 4015 but the print at 4067 doesn't
  gate on it. *(Net.lua:4067 should `and not isReplay`)*
- **Finding (MEDIUM)** — Bot.PickAKANoise (Saudi Master ~3% deceptive AKA)
  fires `S.ApplyAKA + N.SendAKA` at Net.lua:5326-5327 with a suit where the
  bot does NOT hold the boss. The wire flows through `_OnAKA` on remote
  clients → chat hint fires "X (partner) called AKA Y — they hold the boss
  in this suit" → which is **a lie**. The local human partner gets
  misinformation. The whole point of noise AKA is deception of opps; lying
  to your own human partner is a same-team friendly-fire bug.

### Overcall stagger (v2.0.0 BF-20)

- ✓ Random offset 0.4-3.2s within 5s window, leaves 1.8s human breathing room
- **Finding (MEDIUM)** — Bot decision callbacks at Net.lua:1542-1571 do NOT
  gate on `S.s.paused`. If the host pauses mid-overcall window, the C_Timer
  callbacks still fire — `pickOvercall(thisSeat) → RecordOvercallDecision →
  SendOvercallDecision → _OvercallAllDecided → _HostResolveOvercall`.
  The phase guard catches the case where phase changed, but pause within
  PHASE_OVERCALL doesn't change phase — bots silently complete decisions
  during pause. The OVERCALL timeout fn (line 1599 onward) does have a
  pause-aware re-arm, but the staggered bot picks bypass it.
- **Finding (LOW)** — Each bot's offset is independently `0.4 + math.random()
  * 2.8`. Under bad luck, all 4 bots can fire within the first 0.5s,
  re-creating the exact "bots all decided instantly" feel the fix was
  trying to avoid. Stagger should compute slot-based offsets (seat 1 at
  random in [0.4, 1.1], seat 2 in [1.1, 1.8], etc.) for guaranteed
  spread. Acceptable as-is given math.random typical distribution.

## Cross-feature interactions

- **Heartbeat × /baloot reset (host)**: When host runs `/baloot reset` mid-game,
  Slash.lua:248 broadcasts an empty `MSG_LOBBY` so remotes clear their lobby.
  But the host's heartbeat ticker self-stops only when phase becomes IDLE/LOBBY
  — `S.Reset()` at Slash.lua:264 sets phase to IDLE which would self-stop on
  next tick. Idempotent, but during the brief window between Reset and the
  next 15s tick, the host could still emit heartbeats with a now-different
  state. Self-correcting but slightly noisy. **No bug.**
- **AKA chat hint × Drop-recovery**: When a human seat drops and a bot
  replaces them mid-round, if that bot then fires AKA on its first lead, the
  human-partner chat hint correctly fires (the wire is broadcast to all,
  and the bot is no longer the LOCAL player so `fromSelf` doesn't trigger).
  ✓ Works.
- **Settings panel × Slash subcommand drift**: Settings checkboxes only
  cover 9 of the 11+ persistent toggles. Numeric settings + theme cyclers
  are slash-only by design (acknowledged in panel footer). But
  `allowSunOvercall` and `historyEnabled` are not surfaced anywhere user-
  facing — orphaned configs.
- **Stats × SWA Takweesh**: Takweesh-resolution path bumps `contractsTaken`
  for the local-bidder local-seat case (since local was the original
  bidder). It does NOT bump `contractsMade` because `bidderMade` is nil in
  that path. Reasonable — Takweesh outcome doesn't fit the "made/failed"
  binary.

## Bottom line

The new features are mostly well-scoped, but **three structural defects
significantly degrade the value of three flagship v2.0+ fixes**:

1. **MP-21 Heartbeat is half-built** — wire works, no UI consumer. The
   "host gone" warning every player would actually see is missing.
2. **SA-01 Arabic font infrastructure has a broken probe** — `_arabicAvailable`
   is wrongly stuck on True for users without the font, leading to box-glyph
   labels even after the v2.0.2 hotfix was supposed to address this.
3. **BF-30 AKA partner hint is broken for the solo-host case** — the most
   common scenario for the addon's target audience.

Beyond those: BF-11 voice-cue-floor delay (BOT_DELAY_BEL=3.2s) is allocated
but never used; the "Thinking…" indicator misses every escalation rung
(exactly when delays are longest); the Settings panel has a 3.3×-wrong
ISMCTS-budget tooltip; mid-round drop has gaps for SWA-caller-drops and
PREEMPT. None are showstoppers, but for a v3.0.0 architectural release
labeled "ready to ship," the heartbeat + Arabic probe + AKA-host gaps
should be closed.

**Trust verdict for the new surface area: PARTIAL — wire-level
mechanisms work; the user-facing fits-and-finishes have systematic gaps.**
