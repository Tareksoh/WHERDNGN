# v3.0.0 Code Health Audit

## TL;DR
- **Total findings: 28** (Dead 4, Duplication 5, Complexity 7, Comment debt 5, Magic numbers 4, Test gaps 3)
- **Severity:** HIGH 5 / MEDIUM 14 / LOW 9
- **Top 3 highest-impact cleanups:**
  1. **Bot.PickBid is 591 lines** of intermixed shape detection / threshold computation / bidcard hypothesizing / partner-coord blocks across 4+ rounds (R1 R2 sub-branches × Hokm/Sun/Ashkal). Routinely the file's biggest churn target. Two-tier extraction (compute `ctx` once → run round-1 / round-2 sub-pickers) would unlock testability.
  2. **`pickFollow` is 2391 lines** (Bot.lua:4093-6484) with 7+ branch ladders (Faranka, Smother, AKA-receiver, ISMCTS bypass, Sun-pos-4, Tahreeb, must-trump). Every audit fix layered another conditional onto the same flat function instead of extracting a sub-picker.
  3. **`renderActions` is 649 lines** (UI.lua:2025-2659) with one giant `elseif` chain over 8 phases. Each phase block is independently testable; the function as written cannot be unit-tested per-phase without the global S/UI state being primed.
- **Trust verdict:** The codebase is **still maintainable** but has crossed the threshold where **net new audit fixes carry meaningful regression risk**. Three big functions account for ~3700 lines of `Bot.lua` + `UI.lua`. The `~93 fixes` are nearly all surgical conditional inserts. No structural rewrite is needed yet — but the next 3-5 audit waves should focus on extraction and dead-code cleanup, not more inline patches.

A reader trying to understand the current behaviour reads the v0.11.20 block and mentally fights the now-stale 13th-bot-audit context above it.

## Duplication

### CH-10: Bid order `{(d%4)+1, ((d+1)%4)+1, ((d+2)%4)+1, d}` constructed at 5 sites — HIGH
**Where:** Bot.lua:1771, Bot.lua:1808, State.lua:2218, State.lua:2415, UI.lua:2093. Net.lua:2349 also builds `first = (dealer % 4) + 1`.
**What:** Same 4-element array formed inline across three files. Two of the sites (Bot.lua) are inside `PickBid` and build the array twice in the same function (positions 1-4 then again to compute `bidPos`). Extracting `R.BidOrderFromDealer(dealer)` (Rules.lua) returning `{p1,p2,p3,p4}` would deduplicate and centralize the seat-geometry-vs-Saudi-table-convention rule.
**Suggested fix:** Add `R.BidOrderFromDealer(dealer)` and `R.BidPosOf(dealer, seat)` helpers; replace the 5 inline sites.

### CH-11: `(S.NormalizeName and S.NormalizeName(x)) or x` repeated 12+ times — MEDIUM
**Where:** State.lua:636/640/698/699/706, Net.lua:801/835/849/909/981/4126/4200, UI.lua:3501, WHEREDNGN.lua:570/572/585/586.
**What:** The `S.NormalizeName and S.NormalizeName(...) or ...` defensive-existence pattern is exact-string-duplicated 17 times. State.lua:608 is the canonical implementation; `S.NormalizeName` is loaded synchronously at file scope so the existence check is paranoid copy-paste. `S.NormalizeName` is used inside `S.NormalizeName` itself (line 617) without the guard, proving the file-scope guarantee is real.
**Suggested fix:** Add `S.NormName(x)` (or even just inline `S.NormalizeName(x)` since it's a hard dep) — the 4-token guard prefix is 90% of every call site's character count.

### CH-12: Lobby ticker spawn block duplicated between UI.lua:973-985 and Slash.lua:167-176 — MEDIUM
**Where:** UI.lua:973 (Host button) and Slash.lua:170 (`/baloot host`). Both arm a `K.LOBBY_BROADCAST_SEC` `C_Timer.NewTicker` with self-cancellation logic, identical predicate (`isHost and phase == LOBBY`), same shutdown path. Cancel-prior block is also duplicated at UI.lua:657.
**What:** Two entry points, one ticker. If the ticker spawn logic ever changes (cadence, condition, side effects), both sites need the patch. Slash.lua:250/304 also have the cancel-only twin.
**Suggested fix:** Extract `B._StartLobbyTicker()` / `B._StopLobbyTicker()` (e.g. exposed on `B.State` or as `N.HostStartLobbyBroadcast`).

### CH-13: Defensive pcall wrappers around 6 escalation/bid bot dispatchers in `MaybeRunBot` — HIGH
**Where:** Net.lua:4823 (Triple), 4876 (Four), 4927 (Gahwa), 4981 (Preempt), 5107 (Bid), 5171 (Play). All 6 follow the identical pattern: `local applied, skipSent = false, false; local ok, err = pcall(function() ... end); if not ok then log + recovery branch end`. The recovery branches are non-trivial (Triple's at 4844-4858 is 15 lines; Preempt's at 5028-5081 is 54 lines) and each one is uniquely brittle — they re-emit a default decision when an upstream pcall catches a Bot.Pick* error.
**What:** This is a copy-pasted error-recovery skeleton with site-specific variations. Each new escalation rung in v2.x added another 50-line block. The `applied/skipSent` pattern is also copy-paste with subtly-different post-recovery actions per rung.
**Suggested fix:** A `dispatchBotDecision(seat, kind, picker, applyFn, broadcastFn, finalizeFn, recoveryDefault)` helper. Sound design but bigger lift; for now, at least extract the recovery path's "did we fully apply" predicate into a named helper.

### CH-14: Manual `GameTooltip:SetOwner / AddLine / Show / Hide` 13 times in UI.lua — LOW
**Where:** UI.lua:413, 588, 594, 634, 692, 744, 895, 928, 1038, 1134, 1463, 1485, 2005. The `setLobbyTooltip(btn, headline, body)` helper at UI.lua:410 already encapsulates exactly this pattern, AND `addAction(label, onclick, tooltip)` at UI.lua:1980 wraps the same logic with label-color stripping. Yet 13 sites bypass both helpers and inline the 4-call sequence.
**What:** Tooltip styling is now inconsistent — `setLobbyTooltip`, `addAction(..., tooltip)`, and inline calls each do their own thing. Future "tooltip looks weird on this one button" fixes will have N call sites to update.
**Suggested fix:** Migrate the 11 standalone sites (those not on action buttons) to `setLobbyTooltip`. The two embedded inside `addAction` itself are the canonical implementation and are fine.

## Complexity hotspots

### CH-20: `Bot.PickBid` is 591 lines with 4-way branching on `round × bidPos × prior-bid-state` — HIGH
**Where:** Bot.lua:1561-2148.
**What:** The function does (1) auto-Sun-on-4-aces, (2) prior-bid scan, (3) Sun-strength compute with bonuses, (4) Belote detection on hypothetical, (5) urgency computation, (6) threshold derivation with jitter and Bel-fear ramp, (7) position-aware threshold bias, then (8) round-1 sub-tree (Ashkal eligibility with 5 separate gates, direct Sun, Hokm-on-flipped with Belote bypass), then (9) round-2 sub-tree (G-4 partner-Hokm suppression, suit-search, Sun-vs-Hokm 5-margin, BC-MANDATORY Belote). `bidPos` from dealer is computed twice. `withBidcard(hand, S.s.bidCard)` is called 5 times (always with the same inputs).
**Suggested fix:** Extract `_pickBidR1(ctx)` and `_pickBidR2(ctx)` where ctx is the precomputed `{sun, sunHand, belote, hokmHand, urgency, thresholds, bidPos, partnerBid}` block. Will halve cognitive load and let tests target the round-2 search loop directly. Also: hoist `local bidcardHand = withBidcard(hand, S.s.bidCard)` and reuse — eliminates 4 redundant table-clone calls per bid.

### CH-21: `pickFollow` is 2391 lines (Bot.lua:4093-6484) with 9+ flat branches — HIGH
**Where:** Bot.lua:4093.
**What:** Successive audit fixes added: AKA-receiver (4139), implicit-AKA (4144), Faranka pos-4 (4221), 5-factor framework (line 4274 region), Smother / Takbeer, must-trump-ruff, Tahreeb, position-aware following, partner-style-aware following, defender vs bidder asymmetric paths. Every fix wedged a `if Bot.IsM3lm() and contract.type == K.BID_SUN and lastSeat ...` block at the right priority point, growing the function instead of segmenting it. There's no internal structure; reading top-to-bottom is the only navigation.
**Suggested fix:** Extract by trick-position — `_followPos2(...)`, `_followPos3(...)`, `_followPos4(...)`. Each stays large but becomes independently testable. (Caveat: many branches gate on lastSeat / partnerWinning anyway — half the work is already done semantically.)

### CH-22: `renderActions` is 649 lines on one `elseif` ladder over 8 phases — HIGH
**Where:** UI.lua:2025-2659.
**What:** Phases bid (DEAL1/DEAL2BID), DOUBLE, TRIPLE, FOUR, GAHWA, OVERCALL, PREEMPT, DEAL3/PLAY (with sub-branches Takweesh, SWA, AKA, BALOOT), SCORE, GAME_END. The DEAL3/PLAY block alone is 200+ lines. All inline; no per-phase helpers.
**Suggested fix:** Break into `renderBiddingActions`, `renderEscalationActions(phase)`, `renderPlayActions`, `renderSWAResponseActions`, `renderEndgameActions`, dispatch from `renderActions`. (The `addAction` machinery is already the right abstraction; the dispatcher just needs splitting.)

### CH-23: `renderBanner` is 398 lines (UI.lua:3642) with 8+ phase-specific banner styles — MEDIUM
Same pattern as CH-22 but smaller. Each phase's banner shape is unique enough that extracting helpers would be straightforward.

### CH-24: `S.ApplyResyncSnapshot` is 169 lines, `S.ApplyRoundEnd` is 253 lines, `N.HostResolveSWA` is 313 lines, `N.HostResolveTakweesh` is 250 lines — MEDIUM
**Where:** State.lua:432, State.lua:1850; Net.lua:3691, Net.lua:2769.
**What:** Each is a "do every step of one big operation" function. They share a feel: parse inputs → validate → reconstruct related state → mutate → broadcast → side effects. Each grew via accreted audit fixes (HostResolveSWA has 9th-audit-fix, X4-fix, X6-fix, X7-fix, etc.). Cyclomatic complexity is high, but unlike PickBid these are linear pipelines — large but readable top-to-bottom. Less urgent.
**Suggested fix:** Defer. Mark for cleanup-pass after the bigger PickBid/pickFollow/renderActions extraction lands.

### CH-25: WHEREDNGN.lua `OnEvent` PLAYER_LOGIN handler is 360+ lines — MEDIUM
**Where:** WHEREDNGN.lua:155-518.
**What:** PLAYER_LOGIN does ensureDB, init, register prefix, build settings panel (114 lines, 3 sections of checkboxes), welcome print, RestoreSession-and-rearm (170 lines that re-arm 6 different timers + handle 4 different mid-flow recoveries). All inline in one event handler.
**Suggested fix:** Extract `_buildSettingsPanel()` into UI.lua (where it belongs) and `_rearmAfterRestore()` into State.lua/Net.lua. Both are pure compositions of existing functions.

### CH-26: `S.HostAdvanceBidding` round-1/round-2 logic is intermixed at 144 lines — LOW
**Where:** State.lua:2214-2356.
**What:** Round-1 winning-bid resolution (Sun-locks-first, Ashkal eligibility, Hokm-only-if-no-prior) and round-2 winning-bid resolution (Sun-overcalls-Hokm, no-Ashkal, no-flipped-suit) live in the same `for seat in order` loop with `if s.bidRound == 1 then ... else ... end`. Each branch is 30-40 lines. They share little code — only the iteration over `order` and the count.
**Suggested fix:** Defer; not painful enough yet.

## Test gaps

### CH-50: Drop-recovery flow (v1.8.0 + v2.0.0 + v2.2.0) has zero tests — HIGH
**Where:** WHEREDNGN.lua:594-690 implements the GROUP_ROSTER_UPDATE → mid-round bot replace, with synthesized votes for PHASE_DOUBLE/TRIPLE/FOUR/GAHWA/OVERCALL/SWA. None of `tests/test_state_bot.lua` exercises this code (verified: zero matches for `GROUP_ROSTER_UPDATE`, `HostKickSeat` mid-round, `RecordOvercallDecision(seat, "WAIVE")` synthesis from drop, `swaRequest.responses[seat] = true` synthesis).
**What:** This is the most fragile new feature — 6 phase-specific synthesis branches, each making strong assumptions about state at drop time. v2.2.0 already had to add `CancelTurnTimer` (audit MP-33). The next regression here will manifest as a soft-lock in production — exactly the bug class the original v1.8.0 fix was supposed to eliminate.
**Suggested fix:** Add 6 tests — one per drop-during-phase path. Each test sets up the phase, invokes the drop logic with a synthetic dead seat, asserts (a) `seats[seat].isBot == true`, (b) the right vote/decision was synthesized, (c) `MaybeRunBot` advances or `_OvercallAllDecided` triggers.

### CH-51: Settings panel (v3.0 + v2.1.0) checkbox handlers have no tests — MEDIUM
**Where:** WHEREDNGN.lua:176-300 builds a Settings panel with 8+ checkboxes. The OnClick handler at line 219 mutates `WHEREDNGNDB[key]` directly. No test covers the mutation, the OnShow init, or the cascade-effect comment ("clicking M3lm checks Advanced too"). The cascade comment is actually wrong — the OnClick only sets the clicked key; the `Bot.IsM3lm` parent-check happens at read time. The comment claims a write-time cascade that doesn't exist.
**What:** Settings panel is the only persistent-toggle entry point that doesn't go through `/baloot` slash commands (which DO have tests via `test_state_bot.lua` slash flow checks). If a future tier rename or DB-key change happens, this panel silently breaks.
**Suggested fix:** A handful of source-pin tests (matching the existing test style) verifying (a) every persistent key in DEFAULTS has a checkbox, (b) the OnClick body matches the canonical pattern, (c) the cascade comment is accurate or amended.

### CH-52: AKA partner hint banner (v2.0.0 BF-30) and Stats persistence (v2.1.0) coverage limited to source-pin — MEDIUM
**Where:** Net.lua:4760 (partner-bot Bel reason hint), WHEREDNGN.lua:71 (stats subtable). Tests do source-pin the stats schema (no behavioral check that gamesPlayed actually increments after ApplyGameEnd, or that biggestSwing tracks correctly). The AKA partner hint chat-print at Net.lua:4760-4772 has no test at all (it's inside the Triple/Four/Gahwa pcall blocks; not even source-pinned).
**What:** Both are user-visible features. Stats can silently regress (e.g., a future ApplyGameEnd refactor forgets the increment) and no test catches it. The partner-hint chat line could break colour codes or formatting and no test catches it.
**Suggested fix:** Behavioral tests for stats — call `S.ApplyGameEnd("A")` with a primed cumulative, assert `WHEREDNGNDB.stats.gamesPlayed == prior + 1`. For the partner hint, at minimum source-pin the format string + the team-equality guard.

## What's clean

- **Constants.lua is excellent** — every threshold is named with provenance comments, value-history is preserved, the calibration trail is auditable. The `K.*` discipline is the strongest organizational property of the addon.
- **Bot tier dispatch (`Bot.IsAdvanced`/`IsM3lm`/`IsFzloky`/`IsSaudiMaster`)** uses strict-extension parents correctly and cleanly. CLAUDE.md documents the cumulative semantics; the implementations match. Zero confusion in the call sites.
- **Test surface is genuinely large** (~6000 lines test_state_bot.lua + ~3000 test_rules.lua + multi-seed simulators). Source-pin tests are brittle BUT they catch the exact class of regression that audit-driven fix-replacement-fix cycles produce: someone deletes the wrong branch.
- **Wire protocol naming convention** (`K.MSG_*` short tags + canonical Saudi names in player-visible strings) is held consistently. The TAKE_HOKM removal across 7 files (CH-33) shows the repo CAN coordinate cross-file deletions when needed.
- **Defensive `if X and X.Y then X.Y(...) end` calls into optional deps** (B.Bot, B.UI, B.Net, C_Timer) is paranoid but correct — the addon does survive partial-load and test-harness contexts as a result. The cost is verbosity, not bugs.
