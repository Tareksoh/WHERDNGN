# v1.6.1 Saudi Authenticity Audit

Auditor: Agent 5 of 5 (Saudi authenticity angle)
Scope: Does the addon FEEL like a real Saudi Baloot table to a Saudi player?
Method: Walked `Constants.lua`, `UI.lua`, `Slash.lua`, `Sound.lua`, `Bot.lua`,
`docs/strategy/`, the `sounds/` voice-cue inventory, and CHANGELOG history of
prior Arabic/glyph fixes.

## TL;DR

- Total issues: 18 (1 CRITICAL, 6 HIGH, 8 MEDIUM, 3 LOW)
- Severity breakdown:
  - CRITICAL: 1 — **Saudi escalation rungs (Bel/Bel x2/Four/Gahwa) surfaced
    as English "Double x2 / Triple x3 / Four / Gahwa"** in player-visible UI
    despite `CLAUDE.md` policy explicitly mandating Saudi names in
    player-visible text. Direct doc-vs-code violation; the call name a Saudi
    player expects to see ("Bel" / بل) is the *audio* but not the *button*.
  - HIGH: 6 — escalation labels, hardcoded `Belote (K+Q ♥)` (wrong on every
    non-hearts trump), Sun/Hokm rendered LATIN-only with no Saudi-style
    transliteration ("Hokm" / "Sun" instead of حكم/صن or even Saudi
    transliterations like "Sin" or "Sann"), pass-button "Pass" in R1 (the
    real Saudi term is بَسْ which the *audio* says — but UI shows English
    "Pass"), sweep banner says "AL-KABOOT!" (good) but failure says
    "BALOOT!" (Saudi convention is "بلوت!" only on a *successful* contract;
    an addon labeling a *failed* contract "BALOOT!" inverts the cue), match-
    end title `8amt!! go play something else` is shown to BOTH winner and
    loser — Saudi banter says it ONLY to the loser.
  - MEDIUM: 8 — voice cue dialect mix (ElevenLabs Saud .ogg + user-supplied
    .mp3 of varying provenance), bid card label is Latin-only, no Arabic
    name display path, "ASHKAL" all-caps Latin, "Sun-overcall window"
    English jargon, "Round done" generic phrasing, K+Q-of-trump button
    labeled "BALOOT!" (Saudi spelling, good) but the audio file
    (`baloot.mp3`) is shared with the round-end fanfare creating context
    overload, slash command `/baloot saudimaster` is the only tier with a
    Saudi adjective — others (m3lm, fzloky) are Saudi internally but
    surfaced as Latin.
  - LOW: 3 — bot tier names "M3lm" / "Fzloky" / "Saudi Master" inconsistent
    in Saudi-vs-English mix; "WLA" used in overcall but "wla" lowercase in
    bidding phase; suit glyphs are universal cards (♠♥♦♣) — no Saudi-table
    suit-name slang surfaced anywhere (per glossary `سبيت/شريه/هاص/ديمن`
    are documented but not used; this is fine — universal glyphs are
    safer — but worth flagging that the glossary's slang is purely
    documentation, not surfaced).

- Top 3 fixes that would noticeably increase Saudi-feel:
  1. **Re-Saudi-fy the escalation rung labels.** Revert v1.0.2's
     "Bel" → "Double x2" rename. The audio file is `BEL.mp3` saying "بل" /
     "Bel" but the button shows "Double x2"; this is the most jarring
     authenticity gap because the call-name a Saudi player verbally
     expects ("Bel!", "Bel x2!", "Four!", "Gahwa!") is missing from every
     button face. `UI.lua:1843, 1846, 1849, 1851, 1866, 1869, 1882, 1885,
     3253-3256, 3374-3377, 3556-3559`. Use Saudi-transliteration labels
     ("Bel x2", "Bel x3" or "Theri", "Four", "Gahwa") to match what the
     audio says. (CLAUDE.md:56 explicitly mandates this.)
  2. **Fix `Belote (K+Q ♥)` hardcoded hearts glyph.** UI.lua:3262 + 3385
     prints a literal hearts symbol regardless of contract trump. Saudi
     player declaring K+Q-of-spades sees the score banner say "Belote
     (K+Q ♥)" — visually wrong and a direct authenticity bug. Use
     `K.SUIT_GLYPH[trump]`.
  3. **Stop showing `8amt!! go play something else` to the winner.**
     `UI.lua:3148` — title is set unconditionally regardless of who won.
     "غامت" is loser-side Saudi banter; firing it at the winner reads as
     either confusing or condescending. Branch on local-team-vs-winner:
     winners get a celebratory line, losers get the 8amt tease.

## Arabic glyph rendering / font workarounds

### SA-01: No Arabic ever rendered in the UI — addon is fully Latin-trans
- Severity: HIGH (cumulative authenticity drag)
- Cue: A Saudi player opens the addon and sees not a single Arabic
  glyph anywhere — only Latin button labels ("Hokm", "Sun", "Pass",
  "Double x2", "Gahwa"). The auditor's expectation upon seeing "Saudi
  Baloot" is that *some* Arabic appears. The voice cues partially
  carry it, but the visual is 100% Latin.
- Root cause: `Constants.lua:36` pins font to `Fonts\\ARIALN.TTF`
  (Arial Narrow), one of WoW's bundled fonts that does not include
  the Arabic block. Multiple comments throughout `UI.lua` (lines
  1985, 2096-2100, 2456-2462, 3401-3403) document this constraint
  and the team's deliberate fallback to Latin transliteration.
- Fix sketch: WoW *can* render arbitrary fonts via `SetFont(path, ...)`
  with a TTF bundled in the addon (e.g., Noto Naskh Arabic or
  Amiri shipped under `media/fonts/`). The constraint cited in
  source is correct for the *bundled* WoW fonts, but bundling an
  Arabic font with the addon (~300 KB) would unlock real Arabic
  labels. Non-trivial change — flagged as authenticity ceiling.

### SA-02: Title "Loot & Baloot" is English-only
- Severity: LOW
- Cue: The window title is "Loot & Baloot" (`WHEREDNGN.toc:2`).
  Cute pun for English speakers; opaque to a Saudi player who'd
  expect "بلوت" (with optional "Loot &" prefix in Latin).
- Root cause: `WHEREDNGN.toc:2 ## Title: Loot & Baloot`
- Fix sketch: Once an Arabic-capable font is bundled (SA-01),
  optionally append "بلوت" to the title for instant recognition.
  Not strictly necessary; the Latin "Baloot" is the standard
  romanization Saudis use in chat too.

### SA-03: Hardcoded `♥` glyph in Belote score banner
- Severity: HIGH
- Cue: A K+Q-of-spades Belote (Hokm Spades contract) shows
  "Belote (K+Q ♥): Team A +20 raw" in the round-end banner — wrong
  suit. Saudi (and any) player sees an obvious bug.
- Root cause: `UI.lua:3262` and `UI.lua:3385`:
  `("Belote (K+Q ♥): %s +20 raw"):format(teamLabel(r.belote))` —
  literal hearts glyph hardcoded, ignoring the actual trump suit
  in `S.s.contract.trump`.
- Fix sketch: Replace `♥` with `K.SUIT_GLYPH[S.s.contract.trump or "H"]`
  (with a same-color tint via `K.SUIT_COLOR_ONCARD`). 30-second fix.

### SA-04: Suit glyphs are universal cards, not Saudi-table suit slang
- Severity: LOW (informational)
- Cue: The addon uses ♠♥♦♣ everywhere — perfectly readable but no
  evocation of the Saudi-table verbal slang documented in the
  glossary (سبيت / شريه / هاص / ديمن).
- Root cause: `Constants.lua:16 K.SUIT_GLYPH` uses the Unicode card-suit
  block. Glossary at `docs/strategy/glossary.md:347-353` documents Saudi
  slang but it's never surfaced.
- Fix sketch: Universal symbols are safer — DO NOT change. Mentioning
  here only because the glossary's "Card-name slang" + "Suit-name slang"
  sections are pure documentation, not feeding any UI element. Future
  work could add slash-tip flavor lines mentioning sbeed/shareer to
  educate players, but not part of the live addon.

## Voice cue language consistency

### SA-10: Voice pack is split between ElevenLabs Saud (.ogg) and user-supplied vocals (.mp3)
- Severity: MEDIUM
- Cue: A Saudi player listens through a full round and hears:
  voice 1 (auto-gen ElevenLabs Saud-deep) for hokm/sun/ashkal/pass/wla/
  awal/thany/aka, and voice 2 (different speaker, .mp3) for
  bel/three/four/gahwa/khamseen/100/400/sera/baloot. The dialect is
  Saudi for both but the speaker / mic / timbre changes mid-round —
  audible inconsistency in the most identity-defining moments
  (escalation rungs).
- Root cause: `Constants.lua:323-369` mixes `.ogg` and `.mp3` cues with
  comments noting `v1.0.2 user-supplied` for the new ones. Provenance
  trail in `sounds/_make_voice_eleven.py:44` shows ElevenLabs Saud
  (3nav5pHC1EYvWOd5LmnA); .mp3 sources are user-supplied without
  documented speaker identity.
- Fix sketch: Either re-render the ElevenLabs cues in the same voice
  used for the user-supplied .mp3s, OR re-record the .mp3s with the
  same speaker who did the original .oggs. Not a code fix — a content
  fix. Note for the user: "Are these all the same Saudi speaker?
  Sounded different in side-by-side."

### SA-11: `pass.ogg` voice cue uses "بَسْ" but `wla.ogg` uses "ولا" — both correct, but the *button* says "Pass" / "wla"
- Severity: MEDIUM
- Cue: Mismatched audio + visual in round 1. Audio says "بَسْ" (the
  Saudi pass call), button face says "Pass". Round 2 the audio says
  "ولا" and button says "wla" — round 2 matches; round 1 doesn't.
- Root cause: `UI.lua:1719` uses literal `"Pass"` for round 1 and
  `"wla"` for round 2; comment explicitly notes the asymmetry. Audio
  cues `K.SND_VOICE_PASS = "pass.ogg"` carries "بَسْ"
  (`Constants.lua:351`) but the button's Latin label is just "Pass".
- Fix sketch: Change the round-1 button label to `"bas"` (Latin
  transliteration of بَسْ) to match the audio. One-line change at
  `UI.lua:1719`. Also update the side-rendered `bidLabelForSeat`
  (`UI.lua:2463`) to mirror.

### SA-12: No voice cue confirmation for Takweesh / SWA-deny — Saudi tables are vocal
- Severity: LOW
- Cue: Saudi-table Baloot is a verbally noisy game — every escalation,
  every accusation has a vocal cue. The addon has voice for hokm/sun/
  bel/triple/four/gahwa/aka but Takweesh and SWA-deny pass silently
  (no `K.SND_VOICE_*` defined for them).
- Root cause: `Constants.lua:348-369` voice cue list ends at AKA. No
  Takweesh / SWA voice file shipped.
- Fix sketch: Add `K.SND_VOICE_TAKWEESH` (recording: "تكويش!") and
  `K.SND_VOICE_SWA` (recording: "سوا!") files, dispatched in
  `Net.HostResolveTakweesh` and `Net.LocalSWA`. Not blocking — flagged
  as an authenticity-completeness gap.

## Term legibility (Hokm/Sun/escalation rungs)

### SA-20: CRITICAL — Escalation rung buttons use ENGLISH despite CLAUDE.md mandate
- Severity: **CRITICAL**
- Cue: Saudi tables call the escalation rungs by their Saudi names —
  "Bel!" / "Bel x2!" / "Four!" / "Gahwa!" These are the most
  recognizable Saudi-Baloot terms after the bid types themselves.
  The addon's audio cues (`BEL.mp3`, `three.mp3`, `four.mp3`,
  `gahwa.mp3`) carry the Saudi voice — but the *button labels*
  surfaced to the player are "Double x2", "Triple x3 (open)",
  "Four & open (x4)", "Gahwa (match-win)". A Saudi player presses
  a button that says "Double x2" while hearing the audio say "بل!".
  Direct violation of `CLAUDE.md:56`:
  > "Saudi names in docs and player-visible text, code names only inside Lua."
- Root cause: v1.0.2 user-requested rename (CHANGELOG.md:3687-3699)
  flipped Saudi-flavored "Bel"/"Triple"/"Four"/"Gahwa" labels to
  English "Double x2"/"Triple x3"/etc. Affected sites:
  - `UI.lua:1828` `"Double forbidden (Sun >=100)"`
  - `UI.lua:1843, 1846` Sun-Bel button as "Double x2"
  - `UI.lua:1849, 1851` Hokm-Bel buttons as "Double & open/closed"
  - `UI.lua:1866, 1869` Triple buttons as "Triple x3 (open/closed)"
  - `UI.lua:1882, 1885` Four buttons as "Four & open/closed (x4)"
  - `UI.lua:1896` Gahwa button: "Gahwa (match-win)" — only this one
    keeps the Saudi name
  - `UI.lua:3253-3256, 3374-3377, 3556-3559` round-end / contract
    banner mod chips: "Double x2", "Triple", "Four", "Gahwa"
- Fix sketch: Revert to Saudi names in player-visible text. Use:
  - "Bel x2" instead of "Double x2"
  - "Bel x3" or "Theri" (ثري — confirmed in glossary as in active
    Saudi use per video #11) instead of "Triple x3"
  - "Four" stays — already the Saudi loan-word
  - "Gahwa" stays — already correct
  Internal phase / message names (`PHASE_DOUBLE`, `LocalDouble`,
  `MSG_DOUBLE`) MUST stay unchanged per CLAUDE.md guidance — only
  the visible strings flip. Pure UI-string change.

### SA-21: "Sun" and "Hokm" are Latin transliterations only — never the Arabic source
- Severity: HIGH
- Cue: Bid buttons read "Sun" and "Hokm" (e.g., `UI.lua:1735, 1776,
  1794`); seat-bid badge reads "HOKM" / "SUN" / "ASHKAL" all-caps
  (`UI.lua:2466-2474`); contract banner reads "SUN" / "HOKM"
  (`UI.lua:3544-3546`). Latin only — no صن / حكم anywhere visible.
- Root cause: Same Arial Narrow font constraint as SA-01.
- Fix sketch: Once Arabic font is bundled (SA-01), can show e.g.
  "حكم Hokm" or just "حكم". Without that, current Latin-only is the
  best the addon can do — but flagged as the largest single gap
  between the docs (which use Arabic constantly) and the UI (which
  uses Latin entirely).

### SA-22: "Pre-empt" is rendered as `Qablak (Pre-empt)` — half-Saudi
- Severity: LOW
- Cue: `UI.lua:1995` — button text is `"|cff66ddffQablak (Pre-empt)|r"`.
  "Qablak" (قبلك) is the Saudi term and is used; the parenthetical
  English `(Pre-empt)` is helpful but undermines the Saudi-feel.
  Compare: a Saudi player's instinct is to press the button with
  the Saudi name; the English parenthetical is for non-Saudi onlookers.
- Root cause: `UI.lua:1995`.
- Fix sketch: Drop the `(Pre-empt)` parenthetical OR move it to the
  tooltip/confirm-text. Saudi players know what قبلك means.

### SA-23: "wla (waive)" in overcall, "wla" in bidding pass — inconsistent
- Severity: LOW
- Cue: `UI.lua:1936, 1958, 1969` in overcall phase: button reads
  `"|cff999999WLA (waive)|r"` (uppercase + parenthetical).
  `UI.lua:1719, 2463` in bidding phase: lowercase `"wla"` no paren.
  Same Saudi word ("ولا"), three different Latin renderings: `WLA`,
  `wla`, `wla (waive)`. Inconsistent style.
- Root cause: separate code paths added at different times.
- Fix sketch: Pick one canonical form (`wla` lowercase no paren is
  closest to how Saudis actually write it in chat) and apply
  uniformly. Tooltip can carry the English explanation.

### SA-24: Round-end title for SUCCESSFUL contract is "ALLY B3DO" — Saudi-flavored, GOOD
- Severity: (positive — no fix)
- Cue: Banner reads "ALLY B3DO" (`UI.lua:3362`) when the bidder makes.
  "علي بضو" (lit. "On me, b3do" — Saudi card-table win-call) is
  authentic. The banter spirit comes through.
- Root cause: by-design, kept.

### SA-25: Round-end title for FAILED contract is "BALOOT!" — wrong cue
- Severity: HIGH
- Cue: `UI.lua:3358`: when the bidder fails their contract, the banner
  title is `|cffff5544BALOOT!|r contract failed`. Saudi convention:
  the verbal call "بلوت!" is shouted on a *successful* contract make
  (the K+Q-of-trump fanfare) — never on a *failure*. Using "BALOOT!"
  to herald a failure inverts the cue and will read as a bug to a
  Saudi player.
- Root cause: `UI.lua:3358`.
- Fix sketch: Change the failure title to a different Saudi term —
  candidate vocabulary: "خسر" (khasara — "lost"), "ضيع" (dhayya' —
  "wasted/blew it"), or the dramatic "طاح" (taah — "crashed"). The
  successful-bidder branch using "ALLY B3DO" is fine; the failure
  branch needs its own loss-banter line. The `K.SND_BALOOT` audio
  cue is also fired here — should be re-routed to fire only on
  contract-make, not contract-fail (currently the code may be
  conflating these).

## Score format & game-target conventions

### SA-30: Match-end title `8amt!! go play something else` shown to BOTH winner and loser
- Severity: HIGH
- Cue: When the match ends, both teams see the same headline. "غامت"
  is Saudi banter for "blowout / [you] got destroyed". A Saudi
  *winner* expects different language than a Saudi *loser*. Showing
  `8amt!!` to the winner is either confusing (not the right register
  for celebration) or condescending (the winner reads it as aimed
  at them).
- Root cause: `UI.lua:3148` — title is unconditional. The local-team
  WIN/LOST headline is set separately (line 3145 via `setOutcome`)
  but the title beneath shows `8amt!!` to everyone.
- Fix sketch: Branch on `myTeam == winner`: winners get something like
  "|cffffd055يا بطل|r" (ya batal — champion!) or even-more-Saudi
  "|cffffd055خلصها|r" (khallas-ha — finished it); losers get the
  current `8amt!!` tease. The "go play something else" pseudo-aside
  is fine for losers but reads as dismissive when the winner sees it.

### SA-31: Score display "152 / target" — game target is shown but not Saudi-styled
- Severity: LOW
- Cue: `UI.lua:3536-3538`:
  `Team A: |cff66ff66%d|r   Team B: |cffff6666%d|r   /  %d`
  with target=152. Functional but generic. Saudi players expect to
  see "بنط" (banta — point) in score readouts; "152" by itself reads
  as a raw number. Standard Saudi tournament displays show "152
  بنط" or just "152 ب".
- Root cause: `UI.lua:3537-3538`.
- Fix sketch: Append "ب" or "نقطة" suffix once Arabic font is
  available; until then optionally show "152 pts" or just leave bare
  "152" (current).

### SA-32: Contract banner uses `Contract:` Latin label
- Severity: MEDIUM
- Cue: `UI.lua:3565`:
  `|cffaaaaaaContract:|r %s%s  by  |cff66ddff%s|r%s`
  Generic English. Saudi tournament displays use "العقد" / "البيع"
  / "الكونتراط".
- Root cause: `UI.lua:3565`.
- Fix sketch: Drop the `Contract:` prefix entirely — the visual
  context (HOKM ♠ by PlayerName) is self-explanatory; or replace
  with a more compact Saudi-flavored prefix.

### SA-33: Round counter "Round %d" is purely English
- Severity: LOW
- Cue: `UI.lua:3574 ("Round %d"):format(...)`. Generic.
- Root cause: `UI.lua:3574`.
- Fix sketch: "R %d" or "Hand %d" or (with Arabic font) "جولة %d".
  Trivially low priority.

## Suit glyphs & bid-card display

### SA-40: Bid card on the table shows just the card face — no Saudi-table convention overlay
- Severity: MEDIUM (informational)
- Cue: When a player bids "Hokm Spades", the bid card flips face up
  and stays visible (`UI.lua:2712-2716`). Saudi tables typically
  have the bidder verbally re-announce the suit; the addon's audio
  fires the hokm.ogg cue but the bid card itself is just a card —
  no overlay reading "حكم" or "بَسْ شفنا" or anything Saudi-flavored.
  Functional but flat.
- Root cause: `UI.lua:2712-2716` — pure `setCardSlot(slot, S.s.bidCard)`.
- Fix sketch: Optional small text label rendered atop / next to the
  bid card naming the contract verbally ("HOKM ♠" already shown via
  `bidLabelForSeat` at the seat badge — could be promoted near the
  bid card too). Existing data is sufficient; a Saudi player gets
  the info from the audio + seat badge.

### SA-41: `K.SUIT_COLOR_ONCARD` uses 4-color poker deck (S=black, H=red, D=blue, C=green) — non-Saudi
- Severity: LOW
- Cue: The card face uses 4-color suits per `Constants.lua:26-31`
  comment ("four-color-deck poker convention"). This is a poker /
  bridge convention; Saudi tables use the standard 2-color
  (red hearts/diamonds, black spades/clubs).
- Root cause: `Constants.lua:26-31`.
- Fix sketch: This is a deliberate clarity-over-tradition choice (per
  comment). Argument against changing: a Saudi player would
  expect 2-color. Argument for keeping: 4-color is more readable. If
  the user prioritizes Saudi feel over readability, swap diamonds
  back to red. Optional toggle.

## Name display & RTL layout

### SA-50: Arabic player names rendered with same Arial-Narrow font — would show as boxes
- Severity: MEDIUM
- Cue: Saudi WoW servers (especially Word of Warcraft Arabic-region
  guilds) often have Arabic character names. A player named "أحمد"
  joining the lobby would have their name passed through `shortName()`
  (`UI.lua:389-392`) — strips the realm tail — but the rendered
  FontString uses `K.CARD_FONT` = Arial Narrow, which doesn't include
  Arabic. The name renders as boxes / blanks in the seat badges
  (`UI.lua:2508`).
- Root cause: All seat-badge / score-label text uses `K.CARD_FONT`
  (`UI.lua:373`); Arabic glyphs are not in that font.
- Fix sketch: Tied to SA-01. Bundling an Arabic-capable font fixes
  both authenticity and Arabic-name rendering simultaneously. Until
  then, document the limitation.

### SA-51: `shortName` strips by `-` (realm separator) but doesn't handle multi-word Arabic names
- Severity: LOW
- Cue: `UI.lua:391` `(fullName:match("^([^%-]+)") or fullName)`. Strips
  realm. Most WoW characters don't have spaces in names anyway
  (server-enforced), so this is a non-issue in practice.
- Root cause: not actually a bug.
- Fix sketch: None needed.

### SA-52: No RTL handling — but no RTL strings reach the UI either
- Severity: (informational)
- Cue: WoW's text rendering doesn't do bidi shaping. No place in the
  addon mixes Arabic (RTL) and Latin (LTR) in a single FontString
  except in code comments (which never reach the user). So while
  RTL is unsupported, nothing renders RTL — the limitation is
  contained.
- Root cause: by design (Arabic glyphs intentionally avoided).
- Fix sketch: Bundling Arabic font (SA-01) opens this can of worms;
  any mixed-direction string would need explicit ordering. Not a
  current bug.

## Cultural tone / banter

### SA-60: Tone is professional / clean — Saudi card tables are louder
- Severity: MEDIUM (subjective)
- Cue: The error text and tooltips are documentation-clean ("not
  host", "lobby not full", "swap seat 1 ↔ seat 2"). Real Saudi
  Baloot tables have constant verbal banter — teasing, mock outrage,
  approval calls. The addon's voice cues partially carry this (the
  Saud-deep voice has presence) but the text strings are flat
  English.
- Root cause: Clean technical English chosen across `UI.lua` /
  `Slash.lua` for clarity.
- Fix sketch: Add 2-3 banter lines to high-emotion moments:
  - Successful Bel: `"|cffff5544Bel!|r — challenge accepted"` →
    Saudi-flavor: "بل! تحدي" (post-Arabic-font)
  - Failed Gahwa: punchy Saudi loss line
  - Sweep (already has "AL-KABOOT!" — good)
  Minor polish; not blocking.

### SA-61: `Slash.lua` help text is purely English
- Severity: LOW
- Cue: `/baloot help` lists 25+ commands, all English with English
  descriptions. Saudi players would expect at minimum command
  *aliases* in Latin-Saudi shorthand. The codebase already does
  this for some commands — `/baloot ahel`, `/baloot thaleth` aliases
  for `preempt` (`Slash.lua:234`), `m3lm`/`master`, `fzloky`/`signal`,
  `saudimaster`/`master+` — the *alias* path acknowledges the Saudi
  vocab; the visible *help* doesn't.
- Root cause: `Slash.lua:11-37`.
- Fix sketch: Mirror the alias commands in the help output —
  surface the Saudi shorthand alongside the English. Not user-blocking.

### SA-62: AKA banner text "AKA ♥ — PlayerName" — clean but flat
- Severity: LOW
- Cue: `UI.lua:3488` — banner reads
  `"|c<col>AKA|r <suit-glyph> — <name>"`. Functional. AKA is
  pronounced "إكَهْ" by the audio; visually it's just "AKA". Latin-
  flat compared to a Saudi-table call.
- Root cause: `UI.lua:3488`.
- Fix sketch: Once Arabic font available, "إكَهْ" + suit glyph + name.
  Until then, the audio carries the Saudi feel.

  recognizable.
