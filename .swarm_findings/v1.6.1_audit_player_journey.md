# v1.6.1 Player Journey Audit

Audit angle: temporal flow of a player's experience from install to
end-of-game. Focus on comprehension, discoverability, mistake recovery,
and engagement. (Other audit agents cover UI/UX polish, bot feel,
multiplayer QoL, Saudi authenticity.)

## TL;DR

- Total issues: 27 (3 CRITICAL, 9 HIGH, 11 MEDIUM, 4 LOW)
- The addon is **silent on first launch** — no welcome line, no
  popup, no chat hint. The minimap spade icon appears but is the
  only visible artifact. A new player can install, log in, and never
  know the addon is loaded.
- There is **no in-game rules education** anywhere — no `/baloot
  rules`, no help frame, no tooltip explanation of Hokm vs Sun, no
  suit-rank order shown. Saudi-Baloot-specific quirks (9 of trump =
  rank 7, Belote multiplier-immune, escalation chain, AKA semantics,
  SWA permission flow) are taught only in `docs/strategy/` which the
  addon never surfaces.
- The 13+ action buttons that drive the bidding / escalation /
  signal phases (Hokm/Sun/Pass/Ashkal/Bel/Triple/Four/Gahwa/Kawesh/
  AKA/SWA/Takweesh/Baloot/Qablak/Upgrade-to-Sun/Take-as-Sun) have
  **zero hover tooltips**. `addAction` (UI.lua:1672) doesn't even
  attach an OnEnter handler.

### Top 3 fixes for player onboarding

1. **First-launch welcome chat line** — print a single coloured
   banner on initial PLAYER_LOGIN (when `WHEREDNGNDB.welcomed` is
   nil) at WHEREDNGN.lua:130, telling the player the minimap icon
   is theirs and listing `/baloot help`. One-time gate flips the
   flag so reconnects stay quiet.
2. **Tooltip layer for `addAction`** — extend `addAction` at
   UI.lua:1672 to accept an optional tooltip-string argument and
   wire OnEnter/OnLeave like the lobby checkboxes already do
   (UI.lua:849-856). Then pass tooltip text at every call site
   (Hokm/Sun/Ashkal/Bel/Triple/Four/Gahwa/AKA/SWA/Takweesh/Baloot/
   Qablak/Kawesh/Upgrade/Take). One-line change unblocks all 16+
   buttons.
3. **`/baloot help` rules section + missing entries** — Slash.lua's
   help() (lines 10-36) doesn't list `/blt`, `/baloot ismctsdiag`,
   the alias `/baloot help` itself, or any rules cheat-sheet.
   Promised `/baloot swaperm` (referenced in DEFAULTS comment at
   WHEREDNGN.lua:54) is not even implemented in dispatch().

## First launch / discovery

### PJ-01: Silent first launch — no chat indicator that the addon loaded
- **Severity:** HIGH
- **Cue:** The user installs WHEREDNGN, logs in, and sees nothing
  in chat. The spade-glyph minimap icon appears but is unannounced
  (Blizzard players ignore unfamiliar minimap icons by default).
  No popup, no toast, no whisper, no "Type /baloot to begin" hint.
- **Root cause:** `WHEREDNGN.lua` PLAYER_LOGIN handler (lines
  130-340) calls `init()`, registers the addon prefix, shows the
  minimap icon, and optionally does session restore — but never
  emits a single `print()` to chat. Verified by grep: `welcome|
  onboard|tutorial|firstLaunch|hasSeen` returns zero relevant
  hits across the codebase; no `print()` calls fire on PLAYER_LOGIN.
- **Fix sketch:** First-launch one-shot welcome line gated on
  `WHEREDNGNDB.welcomed`. After ensureDB(), if `WHEREDNGNDB.welcomed
  ~= true`, print a one-line greeting (cyan, with the minimap icon
  glyph and a hint about `/baloot help`), set the flag.

### PJ-02: Incoming host announce is silent — pendingHost set but no chat / popup
- **Severity:** HIGH
- **Cue:** A WoW-party member types `/baloot host`. Their addon
  broadcasts MSG_HOST. The recipient's addon silently sets
  `S.s.pendingHost = { name = sender, gameID = ... }` (Net.lua:837)
  with no chat line, no popup, no sound. The "Join" button only
  becomes available if the recipient already has the WHEREDNGN
  window open. A player who hasn't opened the addon UI today has no
  way to know someone is hosting.
- **Root cause:** `N._OnHostAnnounce` (Net.lua:820-840) writes
  pendingHost and a debug log but no user-facing surfacing. Compare
  `Slash.lua:103` which DOES print "no host invite pending" — the
  inverse case is silent.
- **Fix sketch:** When `S.s.pendingHost` flips from nil to a
  populated record (or when the gameID changes), print a chat line
  "|cff66ddffWHEREDNGN|r `<sender>` is hosting — /baloot join or
  click Join in the window". Optionally play a soft chime.
  Suppress repeated prints for the same gameID.

### PJ-03: Minimap icon glyph is a single ♠ — not visibly "Baloot"
- **Severity:** LOW
- **Cue:** New player sees a green-circle button with a small ♠ on
  the minimap rim. Tooltip says "WHEREDNGN / Saudi Baloot" but the
  glyph alone could be misread as guild crest or third-party tool.
- **Root cause:** `MinimapIcon.lua:103` (Unicode ♠ on green).
- **Fix sketch:** Switch to a stack-of-cards TGA, or larger
  bicolor suit glyph. Discoverability hinges on this.

### PJ-04: No interface-options registration — addon invisible from /reload UI ESC menu
- **Severity:** MEDIUM
- **Cue:** Player opens ESC → Interface → AddOns to look for
  WHEREDNGN settings (where most addons live). Nothing is there.
- **Root cause:** No `Settings.RegisterCategory` /
  `InterfaceOptionsFrame_OpenToCategory` calls anywhere (verified
  via grep).
- **Fix sketch:** Register a minimal Interface Options panel that
  shows the difficulty-tier checkboxes, sound toggle, target
  setting, and a "Open WHEREDNGN window" button. Mirror the lobby
  checkbox tooltips for the same options.

## Game setup discovery

### PJ-05: "/baloot host" is the only documented start flow — solo-hosting not surfaced
- **Severity:** MEDIUM
- **Cue:** A player wanting to play vs. bots only doesn't know the
  flow: open window → Host Game → Fill Bots → Start Round. The
  status text at IDLE (UI.lua:2961) says "Idle. /baloot host to
  start." which is fine, but the lobby panel itself doesn't tell
  the user "you can play solo against bots — click Host Game then
  Fill Bots".
- **Root cause:** `lobbyPanel` has no "first-time" instructional
  copy. The 4 seat rows show "(empty)" with no contextual prompt.
- **Fix sketch:** When `S.s.phase == K.PHASE_IDLE` and seat 1 is
  empty, paint a one-line hint above the seat rows: "Press Host
  Game to start a solo or party game. Empty seats can be filled
  with bots."

### PJ-06: Host Game / Fill Bots / Start Round / Join buttons have NO tooltips
- **Severity:** HIGH
- **Cue:** Mouseover on the four primary lobby buttons reveals
  nothing. Player must guess what "Fill Bots" does (does it fill
  ALL empty? Or just one? Will it kick a real player?).
- **Root cause:** `hostStartBtn` (UI.lua:766), `hostNewBtn`
  (UI.lua:777), `fillBotsBtn` (UI.lua:813), `joinBtn` (UI.lua:1031)
  all set OnClick but no `OnEnter`. The 4 difficulty checkboxes
  next to them DO have full tooltips (UI.lua:849-856) — the
  pattern is in the same file but skipped on the headline buttons.
- **Fix sketch:** Add OnEnter/OnLeave + GameTooltip to each — e.g.
  for Fill Bots: "Fill all empty seats with bots. Bot tier set by
  the checkboxes below. Doesn't kick seated humans." Pattern
  already exists at lines 751-760 (swap-buttons).

### PJ-07: `/blt` shorthand never advertised
- **Severity:** LOW
- **Cue:** Power users have to read source to find `/blt` alias for
  `/baloot`. `Slash.lua:401` registers it; `help()` at 10-36 hides
  it. Fix: prepend "commands (or use /blt):" to help().

## Bid-phase comprehension

### PJ-10: Bid card has no label — new player doesn't know what the centre card means
- **Severity:** HIGH
- **Cue:** Round 1 begins. Player sees a 9♥ in the centre of the
  felt and four buttons: Pass / Hokm ♥ / Sun. They have no idea
  why this card is there or that the suit (♥) is the "proposed
  trump" candidate they can adopt with the Hokm button.
- **Root cause:** The bid card is rendered as a raw card face at
  `centerCards.bid` (UI.lua:1625, 2712-2716) with no surrounding
  "BID CARD" caption or "click Hokm to take this suit as trump"
  hint. The status text says "Bidding (round 1) — `<seat>` to act"
  but doesn't link bidder to bid-card.
- **Fix sketch:** Above or next to the bid-card slot during DEAL1
  / DEAL2BID, render a small caption: "Bid card — call Hokm `♥`
  to make hearts trump, Sun for any-suit, or Pass". Localise to
  current state (e.g. round 2 says "any non-flipped suit").

### PJ-11: Action buttons have NO hover tooltips — player must guess each meaning
- **Severity:** CRITICAL
- **Cue:** First time the player sees the bid panel, they see
  buttons "Pass / Hokm ♥ / Sun / Ashkal" (round 1) or "Skip /
  Double x2 / Double & open / Double & closed" (PHASE_DOUBLE) etc.
  No mouseover help, no hint about what each option DOES, no
  warning about consequences. Not just for bidding — every
  escalation / signalling button is in the same boat.
- **Root cause:** `addAction(label, onclick)` at UI.lua:1672 has
  no tooltip parameter. None of the call sites pass tooltip text
  because the function doesn't accept one. Compare the difficulty
  checkboxes (UI.lua:831 `makeBotDifficultyCheck(... tooltip)`)
  which DO accept and wire a tooltip.
- **Affected buttons (UI.lua):** Pass/wla (1720), Hokm `<glyph>`
  (1735), Ashkal (1763), Sun (1776, 1794), Kawesh (1782), R2 Hokm
  (1789), Skip / Double-open / Double-closed (1843, 1849, 1851),
  Triple x3 open/closed (1866, 1869), Four & open/closed (1882,
  1885), Gahwa (1896), Upgrade-to-Sun / WLA-waive (1933, 1936),
  Take-as-Sun (1950), Qablak / Pass (1995, 1998), SWA / Accept /
  Deny (2055, 2079, 2081), TAKWEESH (2041), AKA `<glyph>` (2101),
  BALOOT! (2144), Next Round / New Game (2170, 2179).
- **Fix sketch:** Extend `addAction` / `addConfirmAction` to
  accept an optional `tooltip` parameter. Wire OnEnter/OnLeave
  matching the existing checkbox pattern. Add per-call-site copy
  explaining the rule and Saudi-specific side-effects.

### PJ-12: "Ashkal" button has no rule explanation, Saudi-specific behaviour invisible
- **Severity:** HIGH
- **Cue:** Player at bid position 3 or 4 in round 1 sees an
  "Ashkal" button. No tooltip. Pressing it converts the contract
  to Sun with the partner (not the caller!) as declarer. New
  players will press it expecting "ashkal" = "I'll bid Ashkal (a
  variant)" and be shocked when their partner is suddenly the
  declarer.
- **Root cause:** UI.lua:1763. The code comment at lines 1740-1764
  explains the rule to programmers but the player sees only
  "Ashkal".
- **Fix sketch:** Per PJ-11, attach tooltip "Ashkal — pass the bid
  to your partner who will declare Sun. Only available to bid
  positions 3 and 4."

### PJ-13: "Kawesh" button has no rule explanation
- **Severity:** MEDIUM
- **Cue:** Player with a 5-card hand of only 7/8/9 (rare) sees an
  orange "Kawesh" button appear. Pressing it annuls the round and
  forces a redeal. No tooltip explains what Kawesh is or what it
  does.
- **Root cause:** UI.lua:1782.
- **Fix sketch:** Tooltip "Kawesh — declare a hand of only 7s/8s/9s
  to annul this deal and redeal. Once-per-round opportunity."

### PJ-14: Round-2 "wla" pass label is opaque to non-Arabic readers
- **Severity:** MEDIUM
- **Cue:** Player passes in round 2 — button label is "wla"
  (transliteration of ولا). Without context, looks like a typo of
  "no" or "NLA" or a button name unrelated to passing.
- **Root cause:** UI.lua:1719. Comment says it matches the Saudi-
  table verbal convention ("I have no preference / confirm the
  existing bid") but the player sees "wla" with no tooltip.
- **Fix sketch:** Tooltip on the round-2 pass: "wla (ولا) — pass
  / no preference. Equivalent to Pass in round 1."

## Rule learning / Saudi quirks education

### PJ-20: NO in-game rules primer / cheat sheet exists anywhere
- **Severity:** CRITICAL
- **Cue:** A player who installed WHEREDNGN, opened the lobby,
  filled bots, and started a round has no in-game way to look up:
  - rank order in trump (J highest, then 9, then A, then T, K, Q,
    8, 7) vs non-trump (A, T, K, Q, J, 9, 8, 7)
  - what Belote, Carré, sequence melds are worth
  - that 9 of trump is rank 7 but four 9s are NOT a Carré
  - escalation chain (Bel ×2 → Triple ×3 → Four ×4 → Gahwa
    match-win)
  - Sun ×2 multiplier
  - that bidder needs strict majority to make (tied = bidder fails)
  - last trick = +10
  - SWA permission rule (≤3 cards instant, 4-5 ask, 5+ mandatory)
- **Root cause:** No help frame, no rules panel, no `/baloot
  rules` command. Verified absence: grep
  `helpPanel|RulesPanel|InfoPanel|primerPanel|cheat.?sheet|
  RulesFrame|helpFrame` returns no matches across the codebase.
  All rule docs live in `docs/strategy/*.md` which the addon never
  surfaces. The CLAUDE.md note "9 of trump is rank 7" is a
  programmer-facing hint.
- **Fix sketch:** Add a "?" help button to the main frame top-bar
  (next to Reset / Sound) that opens a 3-tab rules cheat-sheet:
  Tab 1 Rank order + Bid types, Tab 2 Escalation chain + multipliers,
  Tab 3 SWA / AKA / Takweesh / Belote. Plus `/baloot rules` slash
  alias.

### PJ-21: AKA banner appears with no explanation of what AKA means
- **Severity:** HIGH
- **Cue:** Mid-trick, an "AKA `<glyph>` — `<player>`" banner pops
  above the trick area (UI.lua:3469-3490). New player has no idea
  what "AKA" is or what they should do about it (avoid trumping
  partner's high card, follow with low cards in the called suit).
- **Root cause:** `renderAKABanner` writes the text directly with
  no tooltip / no help. The banner has no OnEnter handler.
- **Fix sketch:** Add OnEnter on the AKA banner: tooltip "AKA
  (إكَهْ) — partner signals they hold the highest unplayed card in
  this suit. Don't overtrump them; follow with low when you can."

### PJ-22: BALOOT! button doesn't explain what it does or why it's pulsing
- **Severity:** HIGH
- **Cue:** Bright yellow pulsing "BALOOT!" button suddenly appears
  during play (UI.lua:2144). Player has no idea what it does,
  whether they HAVE to click it, what happens if they don't, or
  why it's pulsing.
- **Root cause:** No tooltip on the button. The pulse is
  attention-grabbing but ambiguous (could read as urgent / scary).
- **Fix sketch:** Tooltip "BALOOT! — you hold both K and Q of
  trump. Click to claim the +20 Belote bonus. Once-per-round
  voluntary; bots auto-announce."

### PJ-23: Sun-overcall window banner shown without explanation
- **Severity:** HIGH
- **Cue:** Player has just bid Hokm. A blue banner appears: "Sun-
  overcall window / 5s left · 0/4 decided" (UI.lua:1376, 1384).
  Player has no idea what's happening — did their bid not work?
  What is the action panel asking them?
- **Root cause:** Banner text is informative for an experienced
  player but opaque to a first-timer. The banner has no OnEnter
  tooltip.
- **Fix sketch:** OnEnter the overcall banner: tooltip "Post-Hokm
  overcall window — bidder may upgrade Hokm→Sun, any seat may take
  the contract as their own Sun, or all may waive (default)."

### PJ-24: SWA banner doesn't explain what SWA is, only the countdown
- **Severity:** HIGH
- **Cue:** Mid-play, a yellow banner appears "SWA — `<player>`
  claims the rest / N cards remaining · auto-approve in Ns"
  (UI.lua:3453). New player has no idea what claiming-the-rest
  means, what Takweesh does, or what Accept / Deny will do.
- **Root cause:** Banner has no OnEnter help. Code comment at
  UI.lua:3411 explains the convention to programmers but not to
  players.
- **Fix sketch:** OnEnter: tooltip "SWA (سوا) — caller claims they
  win all remaining tricks. Accept = grant. Deny = challenge the
  claim (your team penalty if they're right). Takweesh = call it
  illegal. Auto-approve at 0s."

### PJ-25: Trump rank-order quirk (9 second-highest, J highest) never shown
- **Severity:** MEDIUM
- **Cue:** Player makes a Hokm trump = ♥ contract. They see their
  hand sorted with J♥ first, then 9♥, then A♥, then T♥, K♥, Q♥,
  8♥, 7♥. They might not realise this is correct trump rank order
  vs the natural A-K-Q-J they expect. When they later play a J♥
  expecting it to lose to A♥, they're confused when it wins.
- **Root cause:** The card sort uses Saudi trump rank order but
  there's no in-UI explanation of the order anywhere.
- **Fix sketch:** Per PJ-20 rules tab. Plus, show a small "trump
  rank order: J 9 A T K Q 8 7" subtitle when contract type is
  Hokm, anchored under the contract banner. Same for non-trump
  rank order.

## Mistake recovery / error feedback

### PJ-30: Illegal-play warning is a chat-line print, easy to miss
- **Severity:** HIGH
- **Cue:** Player plays a card that violates trick-following or
  trump-ascendancy rules (legal-set border on the card was orange/
  red). The card goes through (Saudi Takweesh rule) but the player
  gets only a chat line "WHEREDNGN warning: this play is illegal
  (`<reason>`). Opponents can call Takweesh." (Net.lua:2524).
  In a busy chat window during a raid or an active group, this
  line scrolls off in seconds and the player doesn't see it.
- **Root cause:** `N.LocalPlay` (Net.lua:2508) uses `print()`. No
  centre-screen banner, no UIErrorsFrame, no audible warning, no
  red flash on the offending card.
- **Fix sketch:** In addition to the chat print, post the same
  message to `UIErrorsFrame:AddMessage(text, 1, 0.3, 0.2, ...)` so
  it appears in the centre-screen warning area. Optionally play
  the warning chime. Optionally pulse the card's border red for
  one second.

### PJ-31: Illegal-card border (orange/red) is not visually distinct enough from "highlighted"
- **Severity:** MEDIUM
- **Cue:** Player sees their hand with most cards bordered gold
  (legal) and one or two bordered orange/red. The colour is the
  only differentiator. Comment at UI.lua:2310 says "Keep full
  opacity so it doesn't look disabled" — but that means a colour-
  blind player or someone glancing quickly may miss the warning.
- **Root cause:** `b:SetBackdropBorderColor(unpack(COL.badEdge))`
  at UI.lua:2311. Only colour signals illegality.
- **Fix sketch:** Add a small red "X" or "!" badge in the corner
  of cards that aren't in legalSet, in addition to the border
  colour. Or fade illegal cards to ~70% alpha so colour-blind
  players see opacity contrast.

### PJ-32: No undo / cancel after a mistaken bid (single-click bids fire instantly)
- **Severity:** MEDIUM
- **Cue:** Player misclicks "Sun" instead of "Pass" and the bid
  fires immediately. Cannot undo — one click and it's broadcast.
- **Root cause:** `addAction(label, onclick)` for bidding buttons
  uses single-click `OnClick` (UI.lua:1690). Only the high-stakes
  escalation buttons (Triple, Four, Gahwa, Takweesh, Deny SWA, Bel
  variants) use `addConfirmAction` (two-click confirm).
- **Fix sketch:** Either:
  - Apply `addConfirmAction` to Sun and Ashkal too (rare bids with
    big consequences), OR
  - Add a 2-second visible "Sun bid sent — undo?" toast that lets
    the player click to retract within the window.

### PJ-33: Hidden-on-ineligible buttons have no "why" tooltip
- **Severity:** LOW
- **Cue:** Sun-after-Sun, AKA-when-no-candidate, etc. resolve by
  hiding buttons (UI.lua:1775, 2092-2104). Player has no feedback
  about why a button isn't there.
- **Fix sketch:** Mirror the "Double forbidden (Sun >=100)" greyed
  + explanation pattern at UI.lua:1828 for other gates.

### PJ-34: `/baloot reset` mid-game has no confirmation (UI button does)
- **Severity:** LOW
- **Cue:** UI Reset uses StaticPopup confirm (UI.lua:514-547);
  slash `/baloot reset` (Slash.lua:115) destroys game state
  immediately with a "reset" chat line. Misclick/muscle-memory
  user wipes their game.
- **Fix sketch:** When `phase ~= IDLE/GAME_END`, require
  `/baloot reset confirm` second-word for the destructive case.

## Engagement loop / stats

### PJ-40: No persistent player stats (lifetime W/L, hands played, biggest win)
- **Severity:** MEDIUM
- **Cue:** Player wins their first game. The "Game over — `<team>`
  wins" banner displays. After they hit New Game, the score
  resets, the previous game's outcome is lost, and there's nowhere
  to look up "I've played 12 games, won 7, my best round was a
  Triple-Four-Gahwa Sun for 800+ points".
- **Root cause:** `WHEREDNGNDB.history` is appended per round
  (State.lua:1949) for telemetry purposes, but no UI surfaces it
  as player stats. The `/baloot history` slash dumps raw rows to
  chat (Slash.lua:267-319), readable only as developer debug. No
  W/L aggregation, no "best round", no "most-Played-against bot".
- **Fix sketch:** Add a "Stats" tab to the lobby panel (visible at
  IDLE / GAME_END only). Show: games played, win count, win-rate,
  longest win streak, biggest single-round delta, most common
  contract (Hokm vs Sun), favourite trump suit. All derivable from
  the existing `WHEREDNGNDB.history` rows.

### PJ-41: No round replay; trick-peek limited to once per hand
- **Severity:** LOW
- **Cue:** Round ends; player can't review "what did Bot2 play in
  trick 3?". Only trick-peek (UI.lua:1255) — once per hand, last
  trick only. `/baloot debug` is a programmer toggle.
- **Fix sketch:** "Show round log" button on round-end banner
  popping a frame with all 8 tricks (lastRoundResult already
  captures the data).

### PJ-42: GAME_END styling identical regardless of how the game ended
- **Severity:** LOW
- **Cue:** Player calls Gahwa, wins outright. Banner identical to
  a normal 152-target win — same chime, same "8amt!!" title
  (UI.lua:3148). No flourish for big-multiplier wins.
- **Fix sketch:** When `lastRoundResult.gahwa/foured/tripled`,
  paint the title brighter gold + extended fanfare.

### PJ-43: Game-end "New Game" is the only path — no "view history" or "change settings"
- **Severity:** MEDIUM
- **Cue:** Game ends. Player sees "New Game" button (UI.lua:2179)
  and the score. Wants to: change bot tier, look at the round-by-
  round breakdown, leave to AFK. The only action is "New Game"
  which immediately resets and re-shows the lobby.
- **Root cause:** UI.lua:2177-2185 only adds one action.
- **Fix sketch:** At GAME_END, also add: "Leave game" (Reset alias),
  "Show history" (per PJ-40), "Settings" (jumps to lobby with
  no game started so settings are editable).

## Slash command & settings discoverability

### PJ-50: `/baloot help` doesn't list `ismctsdiag`, `swaperm` (which is missing entirely)
- **Severity:** MEDIUM
- **Cue:** Power user types `/baloot help`. Output omits:
  - `/blt` — short alias (Slash.lua:401)
  - `/baloot help` itself (Slash.lua:392)
  - `/baloot ismctsdiag` / `ismctsdebug` — ISMCTS budget telemetry
    (Slash.lua:164)
  - `/baloot bidcalc` is listed but its aliases `bidtrace` /
    `biddebug` (Slash.lua:152) are not
  - the option aliases `addbots`, `fillbots`, `master`, `master+`,
    `ismcts`, `signal`, `ahel`, `thaleth`
- **Root cause:** `help()` at Slash.lua:10-36 manually maintained;
  drifted from `dispatch()` reality.
- **Fix sketch:** Reorganise help() to read from a single source-
  of-truth table (command + aliases + description) used by both
  help and dispatch. Or at minimum sweep the missing entries.

### PJ-51: `/baloot swaperm` is REFERENCED but NOT IMPLEMENTED
- **Severity:** HIGH
- **Cue:** WHEREDNGN.lua:54 (DEFAULTS comment) says "Toggle via
  /baloot swaperm if your group always allows without asking."
  Player tries `/baloot swaperm` — output: "unknown command.
  /baloot help".
- **Root cause:** `Slash.lua` `dispatch()` (lines 38-398) has no
  branch matching "swaperm" or "swarequirespermission". Verified
  via `grep -n "swaperm" Slash.lua` returning nothing. The
  `WHEREDNGNDB.swaRequiresPermission` flag is read by Net.lua:3040
  but only manually editable in SavedVariables.
- **Fix sketch:** Add a `swaperm` branch mirroring the `swa`
  toggle (Slash.lua:225-232), flipping
  `WHEREDNGNDB.swaRequiresPermission`.

### PJ-52: No UI affordance for `target`, `swa`, `preempt`, `swaperm`, `cards`, `felt`
- **Severity:** MEDIUM
- **Cue:** Bot-tier flags (`advancedBots`, `m3lmBots`,
  `fzlokyBots`, `saudiMasterBots`) have lobby checkboxes
  (UI.lua:869-905). Cards/Felt have cycle buttons (UI.lua:951,
  1017). But `target` (game-win point cap), `allowSWA` (SWA
  on/off), `preemptOnAce`, `swaRequiresPermission`, `sound`
  (already has top-left) require slash commands. A player who
  doesn't read the slash help has no path to change them.
- **Root cause:** No options panel, no settings tab in lobby.
- **Fix sketch:** Add a Settings tab/panel reachable from the

## What works well

- **Reset is well-protected**: UI Reset button uses a Blizzard
  StaticPopup confirm (UI.lua:514) so a single misclick can't
  destroy a game.
- **Confirm-action pattern for high-stakes buttons**: Triple,
  Four, Gahwa, Takweesh, Deny SWA, Belote variants all use
  `addConfirmAction` (UI.lua:1696, 1843-1898) requiring a second
  click within 2 seconds. Good safety net for irreversible
  high-impact decisions.
- **Bot-tier checkboxes have detailed tooltips** explaining what
  each tier does (UI.lua:869-905). Pattern is excellent and
  should be extended to action buttons (per PJ-11).
- **Status-line context is current and per-phase**: UI.lua:2960-
  3004 emits a phase-aware status string ("Bidding (round 1)",
  "Defenders: Bel? (×2)", etc.). Helps a returning player figure
  out what's happening.
- **Pause overlay has clear sub-text**: "Click |cffffd055>|r to
  resume" for host vs "Waiting for host to resume" for non-host
  (UI.lua:3523-3527). Discoverable + role-aware.
- **`/baloot status` is genuinely useful**: Slash.lua:243-253
  prints phase, isHost, gameID, round, turn, seat list — exactly
  the right level for "what state am I in?".
