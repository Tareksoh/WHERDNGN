## 2. Critical Bugs Requiring Fix

### B-99 — likelyKawesh teammate cross-contamination

**File:** `Bot.lua:365–383`

**Problem:** `Bot.OnPlayObserved` refreshes `mem.likelyKawesh` for the
just-played seat unconditionally, regardless of whether that seat is an
opponent. A for-loop skeleton (`for opp = 1, 4 do ... end` at line 367) with
an empty body and a comment saying "only refresh the flag for opponents"
reveals the intended restriction was never implemented. As written, after any
bot teammate plays three low cards, `likelyKawesh` is set on that teammate's
memory slot, causing `BotMaster`'s sampler to de-weight strong-card pinning
for a seat that is on our own team. This corrupts planning for the rest of the
trick and any subsequent tricks that read the flag.

**Exact patch** (replace lines 365–383):

```lua
    if not wasIllegal then
        local trickNum = #(S.s.tricks or {}) + 1
        if trickNum >= 3 and Bot._memory then
            -- Re-evaluate likelyKawesh ONLY for opponents of `seat`.
            for opp = 1, 4 do
                if opp ~= seat and R.TeamOf(opp) ~= R.TeamOf(seat) then
                    local omem = Bot._memory[opp]
                    if omem then
                        local allLow, anyPlay = true, false
                        for c2 in pairs(omem.played) do
                            anyPlay = true
                            local r = C.Rank(c2)
                            if r ~= "7" and r ~= "8" and r ~= "9" then
                                allLow = false; break
                            end
                        end
                        omem.likelyKawesh = anyPlay and allLow
                    end
                end
            end
        end
    end
```

This moves the `allLow`/`anyPlay` scan inside the opponent loop and reads from
`omem` (the opponent's memory slot) instead of `mem` (the just-played seat's
slot). The existing `mem` variable is still valid for the earlier void-tracking
work above this block.

## 3. High-Priority Concerns

### H-1 — B-61 sunFail direction may be backwards

**File:** `Bot.lua:1578–1582`

When `m.sunFail >= 2`, `PickDouble` raises the Bel threshold by 8, making the
bot *less* likely to call Bel against a repeat defensive-Sun caller. The stated
rationale is that defensive Sun has a low base score so the Bel reward is small.
However, the EV arithmetic cuts the other way: a defensive-Sun bidder is weak.
Weak bidder + our Bel = high probability bidder fails, which means *we* capture
the inflated 2x penalty directly. The threshold increase suppresses a
statistically profitable Bel call. Recommended action: flip the sign (subtract
8 instead of add) and re-run the headless tournament with `sunFail`-heavy seeds
to measure bidder-team win-rate delta before committing.

### H-2 — gahwaFailed counter is dead

**File:** `Bot.lua:183, 234` (written); no read site in any `.lua` file

`gahwaFailed` is initialised in `emptyMemory` (line 183) and incremented in
`Bot.OnRoundEnd` (line 234) but is never consumed by any picker, decision
function, or style query. The comment at line 225 says it is a "reckless
bidder" marker but no code acts on it. This is not a crash risk, but it is
wasted maintenance surface and misleads future contributors. Recommended
action: either add a picker read (e.g. gate Gahwa retaliation on
`gahwaFailed < 1`) or remove the field and the OnRoundEnd branch entirely.

### H-3 — Bot SWA never initiated

**File:** `Net.lua:2029` (`N.LocalSWA`); `Bot.lua` (no call site)

Bots do not call SWA at any point. `N.LocalSWA` (the only entry point for
initiating a claim) is invoked exclusively through the UI action handler
(UI.lua ~line 1703). The auto-accept timer, MSG_SWA_REQ, and the 5-second
window added in v0.4.6 are all functional for human callers, but a bot holding
all remaining winning cards will play them out trick-by-trick rather than
claiming. In long games with bot-heavy tables this inflates round duration
and skips a legitimate strategic tool. Recommended action: add
`Bot.ConsiderSWA(seat)` called from the play-phase dispatch after each
trick end; implement a simple guard (all remaining cards are top-rank or all
same suit as trump that is void in opponents) before calling `N.LocalSWA()`.

## 4. Medium Concerns

### M-1 — C-4 PickDouble 1.0x trump-weight may be liberal

**File:** `Bot.lua:1549–1554`
The Tier 1 fix aligned trump-weight to 1.0x (from 0.5x) to match
`escalationStrength`. A 0.75x middle ground was considered and rejected during
the audit without EV confirmation. The current value structurally enables Bel
on J+9+A trump (~42 pts) which is borderline. No immediate action required but
should be validated against a tournament with forced Hokm contracts before
final release.

### M-2 — PickAKA opt-in undocumented for human callers

**File:** `Bot.lua:1457, 1474`
`PickAKA` suppresses the AKA call when the partner is not a bot
(`Bot.IsBotSeat` check at line 1474). This is deliberate (human partners may
ignore the signal), but there is no UI callout or CHANGELOG entry explaining
the asymmetry. Human-only tables never hear AKA from bot teammates, which some
players find confusing. Add a CHANGELOG note at minimum.

### M-3 — PickPreempt HOKM asymmetry undocumented vs H-11

**File:** `Bot.lua:1720–1724`
The halved penalty/bonus for human partners (line 1720 onward) is internally
consistent but never cross-referenced against the H-11 spec that defines the
bot PASS as a "calibrated weakness signal." A human PASS is treated as
half-strength, which is correct but the reasoning gap creates a maintenance
hazard. Add an inline reference to H-11 in the comment block.

### M-4 — Bot.OnRoundEnd has no idempotence stamp

**File:** `Bot.lua:227–240`
If MSG_ROUND_END were replayed (network retry, late-joiner catchup), both
`gahwaFailed` and `sunFail` would be double-incremented for the same round.
The current net code does not replay this message in practice, but the
defensive stamp costs one field. Add a `_lastRoundId` guard to `OnRoundEnd`.

### M-5 — Headless tournament cannot detect Tier 4 play-heuristic regressions

**File:** `tests/test_state_bot.lua:531–710`
The tournament section runs 30 deals (seeds `i * 7919`) and asserts
`m3lmSum >= basicSum * 0.85`. None of the 30 seeds are crafted to exercise
Tier 4 conditions (trump-drought, likelyKawesh inference, B-77 ace-lead
opportunism, B-82 point-card lead). A regression in any of these heuristics
would be invisible unless the random seeds happen to produce the right
board configuration. Add at least five deterministic seeds specifically
constructed for each Tier 4 condition with expected winner assertions.

## 6. Cross-Cutting Observations

**Headless tournament coverage gap (see M-5 above):** The test harness runs
play-only rounds without exercising the bidding pipeline, Bel/Triple/Gahwa
escalation, Kawesh/Saneen hand-annul, or SWA resolution. Tier 4 heuristics
that fire in edge-case board positions (B-77, B-82, B-99) are statistically
unlikely to trigger across 30 random seeds. The test suite provides a useful
sanity floor but should not be treated as regression coverage for Tier 4 work.

**Late-joiner stale state (latent, not a bug):** The snapshot payload
(Net.lua:~332–414) serialises `s.turn` and `s.turnKind`. If a late-joiner
receives the snapshot mid-trick, their `s.turn` is the seat about to act
next (as documented at line 1046). This is correct and intentional. The
concern raised in Wave 8 was that MSG_PLAY messages arriving before the
snapshot are discarded; this is safe because the snapshot includes all played
cards (`playedCardsThisRound`) and the trick array (`tricks`). No fix needed,
but the serialisation format should be documented in Net.lua as a contract
for future snapshot fields.
