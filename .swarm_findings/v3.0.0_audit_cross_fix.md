# v3.0.0 Cross-Fix Interaction Audit

## TL;DR
- Total interaction issues: 12 (1 CRITICAL, 4 HIGH, 5 MEDIUM, 2 LOW)
- CRITICAL: 1 — Reset popup OnAccept (UI.lua) bypasses MP-71 host-gone broadcast (also bypasses host heartbeat stop)
- HIGH (4): Tier mult missing on escalation rungs (BF-01 vs BF-06/50); FadeBanner re-show during pending Hide (UX-31); Bel cascade voice cue overlap with SND_CONTRACT (no per-cue stagger budget); BALOOT defer fires inside PHASE_SCORE (BF-15 vs round-end audio)
- Top 3:
  1. **CFI-01** (CRITICAL) — `/baloot reset` mid-round + popup branch loses both MP-71 empty-MSG_LOBBY broadcast AND heartbeat stop. Remotes wait 45s for heartbeat-timeout to drop the host.
  2. **CFI-02** (HIGH) — `BOT_DELAY_TRIPLE/FOUR/GAHWA` and `BOT_DELAY_BEL` (preempt) are passed RAW to `C_Timer.After`, never through `botDelay()`. Saudi Master and Basic both pause exactly 3.4/3.7/4.2/3.2s on escalation rungs — BF-01's tier-pacing contract is broken on the most dramatic decisions.
  3. **CFI-04** (HIGH) — `B.UI.FadeBanner` (UX-31) schedules a deferred `b:Hide()` via `C_Timer.After(duration, ...)`. A re-show during the pending hide is a no-op (banner still IsShown), and the deferred Hide fires AFTER the new content was supposed to appear.

## Drop-recovery interactions

### CFI-01: `/baloot reset` popup OnAccept skips MP-71 broadcast and heartbeat stop  (CRITICAL)
- **Scenario:** Host is mid-round (PHASE_PLAY). User types `/baloot reset`. `Slash.lua:230-233` detects host-mid-game and calls `StaticPopup_Show("WHEREDNGN_RESET_CONFIRM")` then `return`s — **the MP-71 empty-MSG_LOBBY broadcast at Slash.lua:243-249 is never reached**. User clicks "Reset" on the popup. The popup's `OnAccept` at `UI.lua:656-674` only does local teardown (cancel ticker, bump _redealGen, CancelTurnTimer, CancelLocalWarn, S.Reset). It NEVER broadcasts the empty MSG_LOBBY teardown ping AND never calls `StopHostHeartbeat`.
- **Issue:** Remotes hold sticky lobby for up to 45s (HOST_HEARTBEAT_TIMEOUT_SEC) waiting for `IsHostLikelyGone()` to flip — exactly the failure mode MP-71 was added to prevent. The heartbeat ticker keeps broadcasting until its self-stop fires next-tick (≤15s) on `phase == PHASE_IDLE`, but that's still behavior the v2.1.0 fix-author probably didn't account for — both the slash command and the UI Reset button drive the same popup, so the popup IS the canonical reset path. The MP-71 fix only covers the bypass-popup branch (already-LOBBY/IDLE/GAME_END phases).
- **Severity:** CRITICAL — defeats the whole point of the v2.1.0 MP-71 ship, on the most common reset path (mid-round).
- **Code references:** `Slash.lua:216-249` (slash reset routes through popup, only emits MP-71 broadcast on the non-popup branch); `UI.lua:652-680` (popup OnAccept does local teardown only, no SendLobby({}, …), no StopHostHeartbeat); `Net.lua:927-958` (remote `_OnLobby`'s allEmpty MP-71 path that won't trigger).
- **Fix:** Move the MP-71 `SendLobby({}, gameID)` broadcast and a `StopHostHeartbeat()` call into `WHEREDNGN_RESET_CONFIRM`'s `OnAccept` body, before `S.Reset()`. Keep the slash-only MP-71 broadcast where it is for the pre-popup branches (lobby/idle/game_end) — but the popup branch is the path that needs the broadcast.

### CFI-A: Lobby kick (MP-41) does not signal kicked client they've been kicked  (MEDIUM)
- **Scenario:** Host in lobby phase clicks "✕" on seat 4 (a human). `UI.lua:917-925` runs `S.HostKickSeat(4)` then `SendLobby(seats, gameID)`. The lobby array now has names[4] = nil/"". Line 822 keeps `s.hostName = seats[1].name`. Line 829 doesn't clear `pendingHost` (gated on `if s.localSeat`). The kicked client is left in `phase == PHASE_LOBBY` with `localSeat == nil`.
- **Issue:** The kicked client has no UI feedback distinguishing "I was kicked" from "I'm watching a lobby I'm not in". They can still interact with the addon (e.g., type `/baloot start` — guarded by isHost, so harmless — but UX is silently confusing). No chat hint, no banner.
- **Severity:** MEDIUM — host has graceful kick, kicked client has no clear "you were kicked" channel. The FIX-FOR (`Constants.lua:30` style chat line "[WHEREDNGN] You were kicked from the lobby.") is one print statement.
- **Code references:** `UI.lua:906-938` (kick button handler — broadcasts only); `State.lua:722-725` (HostKickSeat clears seat record only); `State.lua:770-830` (ApplyLobby on kicked client leaves them in PHASE_LOBBY with nil localSeat).
- **Fix:** In `S.ApplyLobby`, after the seat-iteration loop, detect "I had a localSeat last frame but now don't" and surface a chat line + soft state demotion (e.g., `s.phase = K.PHASE_IDLE`).

### CFI-B: MP-02/03/04 escalation drop calls MaybeRunBot twice  (LOW)
- **Scenario:** Human at seat 2 drops mid-PHASE_FOUR. `WHEREDNGN.lua:608-617` swaps in a placeholder bot. The phase is `PHASE_FOUR`, so the conditional at line 646-656 fires `MaybeRunBot()`. Then line 684-686 fires `MaybeRunBot()` again unconditionally.

## Bot-pacing interactions

### CFI-02: Tier multiplier missing on escalation-rung delays (BF-01 vs BF-06/50)  (HIGH)
- **Scenario:** Saudi Master tier (`tierMult ≈ 1.30`) bot is the bidder. Defender Bels (PHASE_DOUBLE → PHASE_TRIPLE). At `Net.lua:4821`, `C_Timer.After(BOT_DELAY_TRIPLE, ...)` schedules with `BOT_DELAY_TRIPLE = 3.4s` RAW. Same path for `BOT_DELAY_FOUR = 3.7s` at line 4874, `BOT_DELAY_GAHWA = 4.2s` at line 4925, `BOT_DELAY_BEL = 3.2s` for preempt at line 4971.
- **Issue:** The v1.8.0 BF-01 fix introduced `botDelay(base, difficulty)` that applies tier mult + difficulty offset. BF-01 was wired at `Net.lua:5103` (`botDelay(BOT_DELAY_BID, ...)`) and `5163` (`botDelay(BOT_DELAY_PLAY, ...)`). v2.0.0 BF-06/50 then ADDED per-rung constants for Triple/Four/Gahwa but used `C_Timer.After(BOT_DELAY_TRIPLE, ...)` directly — bypassing `botDelay()`. Net effect: tier-pacing variance is intact for bid+play but completely flat for the most narrative escalation moments. Saudi Master and Basic both fire Triple at 3.4s exactly; the BF-01 design intent ("Saudi pros pause longer on terminal rungs") is lost on the rungs that matter most.
- **Severity:** HIGH — the most visible bot-feel moments (escalation drama) have no tier signature. Two layered fixes silently disagreed on the contract.
- **Code references:** `Net.lua:4821, 4874, 4925, 4971` (raw BOT_DELAY_* constants); `Net.lua:4666-4672` (`botDelay` definition); `Net.lua:5103, 5163` (correct `botDelay` wrap on bid/play).
- **Fix:** Wrap each `C_Timer.After(BOT_DELAY_*, ...)` at lines 4821/4874/4925/4971 with `botDelay(BOT_DELAY_*, "normal")` (or a tier-aware difficulty classifier per rung — e.g., Gahwa is always "hard" because match-stake).

### CFI-03: Bel cascade fires synchronously inside MaybeRunBot — voice cue overlaps SND_CONTRACT  (HIGH)
- **Scenario:** R1 bidding ends. `_HostStepBid` calls `S.ApplyContract` which plays `SND_CONTRACT` (chime, ~0.10s class) AND immediately calls `MaybeRunBot()` per existing flow. Inside MaybeRunBot's PHASE_DOUBLE branch (`Net.lua:4715-4811`), `B.Bot.PickDouble(belSeat)` is called **synchronously** at line 4742. If a bot defender Bels, `S.ApplyDouble` fires `SND_VOICE_DOUBLE` (voice, 0.80s class) at `State.lua:1224` — **in the same Lua tick as SND_CONTRACT**. The 3.2s `BOT_DELAY_BEL` floor (BF-11) is never honored for the bot's OWN Bel decision; it's only honored for the CHAINED Triple/Four/Gahwa.
- **Issue:** The BF-11 voice-cue floor was specifically about giving voice cues space. But the FIRST Bel in a Bel-cascade (defender Bel, PHASE_DOUBLE) decides synchronously — BF-11's 3.2s floor doesn't gate it. The contract chime + Bel voice land on top of each other. Worse, the next rung (PHASE_TRIPLE) waits 3.4s, so the cascade looks like: t=0 SND_CONTRACT + SND_VOICE_DOUBLE simultaneously, then 3.4s gap to SND_VOICE_TRIPLE. Asymmetric pacing.
- **Severity:** HIGH — undermines the v1.8.0 BF-11 voice-floor fix on the cascade's first beat. Reported user-perception class ("voice cues stack mid-word" from `bot_feel.md:33`) is not fully closed.
- **Code references:** `Net.lua:4715-4811` (synchronous PickDouble + ApplyDouble); `State.lua:1188, 1224` (back-to-back SND_CONTRACT then SND_VOICE_DOUBLE).
- **Fix:** Wrap the first-Bel decision in a `C_Timer.After(BOT_DELAY_BEL, ...)` mirroring the Triple/Four/Gahwa pattern. BF-11's design intent was the cue spacing; this site missed it.

### CFI-08: BF-15 BALOOT defer can fire DURING PHASE_SCORE round-end audio  (HIGH)
- **Scenario:** A bot's K-of-trump play closes the K+Q pair AND wins trick 8 (last trick of round). `_HostStepPlay` resolves the trick, transitions to PHASE_SCORE, plays round-end fanfare (e.g., SND_HOKM_LOST / SND_KABOOT / SND_BALOOT for sweeps). Meanwhile, `N._HostMaybeAutoBelote` was called at line 5348 just before `_HostStepPlay`. The K+Q-pair closure scheduled `C_Timer.After(1.0, ...)` at `Net.lua:270-279` — fires 1.0s later, INSIDE PHASE_SCORE. The closure only checks `if not S.s.contract or S.s.contract.type ~= K.BID_HOKM` — but `s.contract` is NOT cleared at PHASE_SCORE (only at next round's `ApplyStart` or full Reset). So the closure proceeds: `S.ApplyBeloteAnnounce` → `B.Sound.Try(K.SND_BALOOT)`.
- **Issue:** SND_BALOOT (the +20 K+Q-pair Belote-announcement chime) overlaps with the round-end banner audio it was specifically deferred AWAY from. The BF-15 1.0s delay was supposed to space the play SFX — it instead lands right inside the round-end fanfare beat. The SFX_INTERVAL=0.10 throttle doesn't help (different soundIds).
- **Severity:** HIGH — the BF-15 fix INVERTED the timing on the very edge case (K+Q closing trick 8) it was supposed to clarify.
- **Code references:** `Net.lua:259-285` (deferred AutoBelote — checks contract not phase); `State.lua:1303-1309` (ApplyBeloteAnnounce plays SND_BALOOT); `State.lua:1850-1867` (ApplyRoundEnd does not clear s.contract).
- **Fix:** Add `if S.s.phase ~= K.PHASE_PLAY then return end` to the deferred closure at `Net.lua:271-276`. If the round ended during the 1.0s defer, the BALOOT cue should be skipped (the round-end fanfare carries the closure).

### CFI-05: v0.8 client TAKE_HOKM silently fails RecordOvercallDecision; force-WAIVED on timeout  (MEDIUM)
- **Scenario:** A v0.8 client (still in the wild, see CHANGELOG v1.5.3) connects to a v3.0.0 host. Overcall window opens; v0.8 client clicks "Take as Hokm-of-Spades" and emits `MSG_OVERCALL_DECISION;{seat};TAKE_HOKM_S`. v3.0.0 host's `_OnOvercallDecision` (Net.lua:1436) calls `S.RecordOvercallDecision`. Per v1.5.3 silent-reject (State.lua:1076-1079), the validation rejects everything except `UPGRADE`/`TAKE`/`WAIVE`. The host doesn't record the v0.8 client's decision, doesn't broadcast it. v0.8 client thinks they took the contract; host's `_OvercallAllDecided` returns false (one seat undecided). 5s window expires; `overcallTimerFn` (Net.lua:1599-1620) fires `_HostResolveOvercall` with the v0.8 client implicitly WAIVE'd.
- **Issue:** v0.8 client thought they took the contract; result is the original bidder's contract goes through. v0.8 client UI may briefly show "I'm bidder" then resync to the real outcome. MP-60's chat warning (since v2.0.0) at `Net.lua:851-865` does fire on first MSG_HOST exchange — so the v0.8 player has been told they're version-mismatched. They've been warned, but the silent-take-becomes-WAIVE is still UX-confusing on the affected hand.
- **Severity:** MEDIUM — softer than crash (no crash thanks to `_HostResolveOvercall`'s default-WAIVE on missing decisions) but the silent semantic flip is exactly what the v1.5.3 silent-reject tradeoff accepts. Since v3.0.0 layered MSG_HEARTBEAT (also unknown to v0.8), a v0.8 host with v3.0.0 clients ALSO gets unknown-tag drops on HEARTBEAT (handler default-falls-through, no error log).
- **Code references:** `State.lua:1068-1085` (RecordOvercallDecision rejects TAKE_HOKM_*); `Net.lua:1436-1467` (`_OnOvercallDecision`); `Net.lua:725-728` (`MSG_HEARTBEAT` handler); `Net.lua:843-865` (MP-60 mismatch warning fires once, on MSG_HOST).
- **Fix:** Optional: when host detects a v0.8-class peer at lobby time (via `peerVersions`), refuse Start until peer reloads. Hard reject is harsher but closes the silent-flip class.

## UI render-loop interactions

### CFI-04: FadeBanner re-show during pending Hide is a no-op; pending Hide kills the new content  (HIGH)
- **Scenario:** AKA call ends — `renderAKABanner` is called (Refresh). Old call cleared, falls into the fade-out branch at `UI.lua:4101-4103`: `B.UI.FadeBanner(b, 0.25, true)`. `fadeBanner` body at `UI.lua:480-493` calls `UIFrameFadeOut` and schedules `C_Timer.After(0.25, function() b:Hide() end)`. Within 0.25s, a new AKA call comes in via `_OnAKA`. Refresh fires renderAKABanner, content updates, and falls into fade-in branch at `UI.lua:4119`: `B.UI.FadeBanner(b, 0.25)`. Inside fadeBanner, line 495 checks `if not b:IsShown()` — banner IS still shown (Hide hasn't fired yet, the deferred timer is still pending). Branch is no-op. The banner content has been updated for the new AKA but the fade-in machinery doesn't re-engage. 0.25s after the original fade-out started, the deferred `b:Hide()` fires — hiding the banner that should be showing the new AKA.
- **Issue:** Classic race: re-show during pending fade-out is silently dropped, and the pending Hide kills the new content. The banner blinks and disappears for the new AKA call.
- **Severity:** HIGH — visible UX bug on rapid AKA cycling (mid-trick partner signal exchanges).
- **Code references:** `UI.lua:477-505` (fadeBanner body); `UI.lua:4101-4120` (renderAKABanner fade-out then fade-in pattern); same anti-pattern usable from any Refresh-driven banner.
- **Fix:** Track a `b._fadeGen` token on each `fadeBanner` call. The deferred `Hide` closure captures the gen at scheduling time; if it doesn't match `b._fadeGen` at fire time, skip Hide. The fade-in branch should bump the gen and force-cancel any in-flight `UIFrameFadeOut` chain via `UIFrameFlash`-style stop or set alpha directly.

### CFI-07: Confirm-arm 2s timer survives clearActions; setText/SetWidth on hidden button  (LOW)

## Bottom line

The v3.0.0 codebase has 1 CRITICAL cross-fix interaction (CFI-01, the popup/MP-71/heartbeat triple-bypass) that defeats the intended fix on the most common reset path. The CRITICAL is also one-edit-fix and very narrow.

The 4 HIGH issues are all "fix N's contract was silently broken by fix N+1": tier-mult missing on escalation rungs (BF-01 vs BF-06/50, CFI-02), Bel-cascade voice cue overlap (BF-11 vs synchronous PickDouble, CFI-03), FadeBanner re-show race (UX-31 self-conflict, CFI-04), and BALOOT defer firing inside PHASE_SCORE (BF-15 vs round-end audio, CFI-08).
