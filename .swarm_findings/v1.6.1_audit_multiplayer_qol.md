# v1.6.1 Multiplayer Quality-of-Life Audit

## TL;DR

- Total issues: 19
- CRITICAL: 2 (game becomes unrecoverable)
- HIGH: 7 (frequently encountered, painful)
- MEDIUM: 7
- LOW: 3 (cosmetic / rare-edge)

Top-3 fixes that would meaningfully improve real-game experience:

1. **MP-04 (CRITICAL)** — Player drop mid-round leaves a dead seat (host
   nulls the seat in `WHEREDNGN.lua:392`, but no bot replacement → AFK
   timer no-ops, MaybeRunBot no-ops, the trick is permanently stuck).
   Single highest-impact fix: have `GROUP_ROSTER_UPDATE` swap the kicked
   seat to a bot mid-game, not just delete it.
2. **MP-21 (CRITICAL)** — No host-transfer logic anywhere. If host's
   WoW client crashes / they /reload offline / leave the party, the
   entire session dies silently for the other 3 players (no "host gone"
   surfaced; their UI just freezes on the last broadcast frame). Add a
   heartbeat-driven dead-host detection + lobby surface.
3. **MP-12 (HIGH)** — `S.HostHandleJoin` (`State.lua:683-698`) does raw
   `==` equality on names without `S.NormalizeName`, so a same-realm
   re-join after /reload can double-seat or fail-to-seat depending on
   which side has the realm suffix. Wrap both equality checks with the
   normalizer that's used everywhere else.

## Disconnect / drop scenarios

### MP-01: Mid-round player drop leaves a dead human seat (no bot replacement)
- **Severity:** CRITICAL
- **Symptom:** A human player drops from the party (DC, /reloadui that
  removes them from group, they leave manually) during a live round.
  The host's `GROUP_ROSTER_UPDATE` handler nulls the seat. Trick play
  is now waiting on `s.turn = <empty seat>` — but `StartTurnTimer`
  early-returns at `if not info or info.isBot then return end`
  (`Net.lua:4127`) so no AFK fires. `MaybeRunBot` also no-ops (`isBotSeat`
  returns falsy for nil seat). The table sits frozen forever.
- **Root cause:** `WHEREDNGN.lua:381-397` calls `S.HostKickSeat(seat)`
  which sets `s.seats[seat] = nil` (`State.lua:701-704`). Nothing
  re-fills the seat with a bot, nothing advances the turn.
- **Repro:** 4-human game; one player force-quits WoW or removes
  themselves from party while it's their turn to play / bid.
- **File:** `WHEREDNGN.lua:392`, `State.lua:701-704`
- **Suggested fix sketch:** In `GROUP_ROSTER_UPDATE`, detect mid-game
  drops and replace with a bot stub (`S.HostKickSeat` → `S.HostReplaceWithBot`
  that re-uses the dropped seat number with `isBot=true`). Then re-fire
  `MaybeRunBot()` if the dropped seat held the active turn / belPending /
  preemptEligible.

### MP-02: Drop during PHASE_DOUBLE / TRIPLE / FOUR / GAHWA window soft-locks
- **Severity:** CRITICAL
- **Symptom:** Same as MP-01 but specific to the escalation windows.
  Bel/Triple/Four/Gahwa each await a SPECIFIC seat. If that seat drops,
  `StartBelTimer`'s seat-info gate (line 4212-4213) returns silently
  with no timer armed. Round soft-locked.
- **Root cause:** `StartBelTimer` at `Net.lua:4207-4217` checks
  `if not info or info.isBot then return end` — when `info` is nil
  after roster-kick, no recovery. The "either-defender" Bel logic
  (`MaybeRunBot:4337-4419`) iterates `belPending` but only checks
  `isBotSeat(seat)`; if the seat is gone, it falls through to the
  human branch and arms a timer that immediately bails.
- **Repro:** 4-human game; right after contract is announced, one of
  the two defenders DCs. The Bel window has nobody to wait on; no
  auto-skip; no fallback.
- **File:** `Net.lua:4207-4217`, `Net.lua:4337-4419`
- **Suggested fix sketch:** In `MaybeRunBot`, treat empty seats in
  `belPending` / Triple/Four/Gahwa eligibility as "auto-skip" — emit
  the appropriate `MSG_SKIP_*` and continue. Same logic for
  `preemptEligible`.

### MP-03: Drop during PHASE_OVERCALL drops a vote, defaults silent
- **Severity:** HIGH
- **Symptom:** Sun-overcall window is open (5s); a peer drops mid-window.
  Their `MSG_OVERCALL_DECISION` never arrives. Window expires on the
  5s timeout (`_HostResolveOvercall`), but the resolution treats the
  missing seat as having waived (no opt-in present) — usually correct
  but worth surfacing as "X dropped, defaulted to waive".
- **Root cause:** No drop-detection inside the overcall window;
  `_OvercallAllDecided` only counts present decisions and the timer
  fires by default. Functionally fine, but silent.
- **Repro:** Player drops within the 5-second overcall window.
- **File:** `Net.lua:1349-1380`, `Net.lua:1382-1454`
- **Suggested fix sketch:** When `GROUP_ROSTER_UPDATE` detects a drop,
  if `S.s.phase == K.PHASE_OVERCALL`, treat their vote as WAIVE and
  re-evaluate `_OvercallAllDecided`. Surface a chat message
  "<name> dropped during overcall — counted as waive".

### MP-04: Drop during SWA permission window leaves vote pending
- **Severity:** HIGH
- **Symptom:** Caller asks for SWA permission (4+ cards). One opponent
  drops before responding. The SWA request body in `S.s.swaRequest.responses`
  never gets their response. The request is held until the 5s timer
  in `Net.lua:3094-3120` fires auto-approve, OR — if /reload re-arms
  it (`WHEREDNGN.lua:314-336`) — even longer.
- **Root cause:** `_OnSWAResp` never fires for a dropped seat; auto-
  approve eventually fires, but in the meantime `MaybeRunBot` is
  blocked (`Net.lua:4313-4319`). Net result: 5-10 seconds of frozen
  table even with the auto-approve.
- **Repro:** Caller invokes SWA with 4+ cards; opponent on the
  blocking team disconnects within 5s.
- **File:** `Net.lua:3315-3393`
- **Suggested fix sketch:** In `GROUP_ROSTER_UPDATE`, if a seat in
  `S.s.swaRequest.responses` lookup is dropped, synthesize an `accept`
  for them (or call `_OnSWAResp` with `__host__` sender on their
  behalf). Cuts the wait from 5s to instant.

### MP-05: Bot-seat does not absorb drops; bot delegation depends on s.seats[seat].isBot
- **Severity:** MEDIUM
- **Symptom:** Even if MP-01 is fixed by replacing seat with bot, the
  bot's `_partnerStyle` ledger for that seat (Bot.lua) accumulated
  from the ex-human's actual plays. New bot inherits the human's
  pattern history — could be wildly off the bot's natural style and
  produce immediate "this bot just changed personality" tells visible
  to the other 3 players.
- **Root cause:** `Bot._partnerStyle[seat]` (Bot.lua:252-389) is
  seat-keyed, no concept of "this seat used to be human". A
  replacement bot reading its own seat's ledger sees plays it didn't
  make.
- **Repro:** Replace dropped human with bot mid-round (post-MP-01 fix).
  Bot reads its `_partnerStyle[seat].leadCount[suit]` and biases its
  next decision based on the human's earlier behavior.
- **File:** `Bot.lua:252-389`
- **Suggested fix sketch:** When swapping a seat from human → bot,
  zero `Bot._partnerStyle[seat]` and `Bot._memory[seat]` so the new
  bot gets a clean read on opponents (without erasing the WHOLE
  ledger).

## /reload handling

### MP-10: Player /reload during PHASE_DEAL2BID with stuck redeal — fixed (validated)
- Confirmed mature handling at `WHEREDNGN.lua:247-261` per v0.10.6/v0.11.0
  RT07-01. Pre-v0.11.0 this was destructive; now `s.redealing` survives
  `/reload` in `TRANSIENT_FIELDS` (intentionally) and `_HostExecuteRedeal`
  re-fires after a 3s recovery delay.

### MP-11: /reload mid-overcall on host loses the per-seat decisions
- **Severity:** MEDIUM
- **Symptom:** Host /reloads at second 2 of a 5-second Sun-overcall
  window. Host re-arms a fresh 5-sec timer (`WHEREDNGN.lua:300-313`),
  but the previously-recorded decisions live in `S.s.overcall.decisions`
  which IS persisted. So decisions survive — but the new 5-second window
  is 5s (full reset), not 3s remaining. Peers get a confusing "wait,
  why is the timer back?" experience.
- **Root cause:** The re-arm at `WHEREDNGN.lua:304` does
  `S.s.overcall.startedAt = (GetTime and GetTime()) or 0` — explicit
  full-reset to current time. By design (per the v0.9.0 M2 comment).
- **Repro:** Open Sun-overcall window; host /reloads at second 2.
  Peers see the 5-second clock magically jump back to 5s.
- **File:** `WHEREDNGN.lua:300-313`
- **Suggested fix sketch:** Either send a fresh `MSG_OVERCALL_OPEN`
  on host /reload so peers also reset their displayed countdown,
  OR (less invasive) chat-print "Host reloaded — overcall window
  extended" so the time-warp is explained. Currently silent.

### MP-12: /reload during host's mid-trick 4-card resolution skipped briefly — handled
- Validated at `WHEREDNGN.lua:218-237`: 0.5s delayed re-fire of
  `_HostStepPlay` if a 4-play trick was on the table. Good.

### MP-13: Non-host /reload mid-round triggers resync but window is short
- **Severity:** LOW
- **Symptom:** Non-host /reloads. `PLAYER_ENTERING_WORLD` schedules
  `maybeRequestResync` 2.0s later (`WHEREDNGN.lua:355`). If the WoW
  party hasn't reformed by 2s, `IsInGroup()` returns false and the
  resync is skipped permanently. Remote player sits with stale
  RestoreSession state.
- **Root cause:** Single-shot resync at fixed 2.0s. No retry on
  `GROUP_ROSTER_UPDATE` when group eventually appears.
- **Repro:** /reload while WoW is loading slowly or the party
  reforms more than 2s after `PLAYER_ENTERING_WORLD`.
- **File:** `WHEREDNGN.lua:100-113`, `WHEREDNGN.lua:355`
- **Suggested fix sketch:** Retry-on-roster-update once: if
  `WHEREDNGNDB.lastGameID` is set and we haven't yet received a valid
  `MSG_RESYNC_RES` and the group becomes available, fire `SendResyncReq`
  again (gated by the existing 30s expectingResyncRes window).

### MP-14: /reload mid-PHASE_PREEMPT recovers only first eligible human's timer
- **Severity:** LOW
- **Symptom:** Pre-emption window (Triple-on-Ace, الثالث) has multiple
  eligible seats. Host /reloads. The recovery code at
  `WHEREDNGN.lua:209-217` arms a `BelTimer` only for the FIRST eligible
  human in the list (`break` after first match). Subsequent eligible
  humans rely on `MaybeRunBot` being re-pumped after the first
  resolves — works in practice but the comment at line 211-216
  acknowledges "only one timer can be in flight". This is a known
  simplification.
- **Repro:** 3 eligible human pre-empters, host /reloads while window
  is open. First (lowest-seat-order) gets a timer; others wait.
- **File:** `WHEREDNGN.lua:209-217`
- **Suggested fix sketch:** Acceptable as-is given the "one window per
  seat in sequence" architecture. Document that the chain is
  serialized so readers don't expect parallel timers.

## Host transfer / crash

### MP-21: No host-transfer logic, no host-alive heartbeat
- **Severity:** CRITICAL
- **Symptom:** Host's WoW client crashes (game closed, system hang,
  router unplugged). Other 3 players' state freezes on the last
  broadcast frame. No surface message. They eventually realize
  via `GROUP_ROSTER_UPDATE` (host left party) but their UI doesn't
  call out "host gone — game over"; the lobby panel just shows seat 1
  as host with a stale name. The 3 remaining players cannot recover —
  no way to elect a new host without `/baloot reset`-ing.
- **Root cause:** No `MSG_HEARTBEAT` from the host. No timeout-based
  detection. The host's authority is implicit; there's no fallback.
  Comment at `Net.lua:10582` of CHANGELOG explicitly admits:
  "All-4-disconnect: non-host state lost (no resync mechanism after
  game-end)".
- **Repro:** Host force-quits WoW mid-round; observe the 3 remaining
  players' UIs.
- **File:** Architectural — affects all of `Net.lua` host-broadcast
  paths. No single line; missing feature entirely.
- **Suggested fix sketch:** Either (a) add a 30-second host
  heartbeat broadcast (`MSG_HOST_ALIVE`) that, when missed for 60s,
  surfaces a "Host appears offline — /baloot reset to start fresh"
  banner on every client; or (b) implement seat-1-of-remaining
  host-transfer where the lowest-seat surviving human auto-promotes
  to host and re-broadcasts `MSG_LOBBY` from their snapshot.
  Option (a) is simpler and lower-risk — host transfer is non-trivial
  given the host's authoritative role across all wire validation.

### MP-22: Host /baloot reset mid-round wipes other players silently
- **Severity:** HIGH
- **Symptom:** Host runs `/baloot reset` mid-round. Their state goes
  to IDLE. They stop broadcasting. Remote players see no `MSG_GAMEEND`,
  no termination signal — the table just freezes. They end up calling
  `/baloot reset` themselves to recover.
- **Root cause:** `Slash.lua:115-135` resets local state but emits no
  termination wire. The "game over" frame never fires; remote clients
  remain in the live phase, waiting indefinitely.
- **Repro:** Host runs `/baloot reset` while peers are mid-trick.
- **File:** `Slash.lua:115-135`
- **Suggested fix sketch:** Have host's reset (when there's an
  active gameID) broadcast a synthesized `MSG_GAMEEND;ABORT` first so
  peers exit cleanly to PHASE_GAME_END or PHASE_IDLE.

### MP-23: Host-name resolution at /reload uses last-writer-wins from announce
- **Severity:** MEDIUM
- **Symptom:** Two players in the same WoW party both run `/baloot host`
  before any peer accepts (e.g., a 5-player party where two race to
  host). Each spams `MSG_HOST` every 3s (`K.LOBBY_BROADCAST_SEC`).
  Victim peer's `pendingHost` is overwritten on each reception. Whichever
  host's announcement landed last "wins", which the victim joins. The
  losing host stays in PHASE_LOBBY indefinitely.
- **Root cause:** `Net.lua:837` writes `S.s.pendingHost` unconditionally.
  Already documented in the v0.10.2 review (`B-Net-10_lobby_host_join.md`
  line 317-365); no fix shipped.
- **Repro:** Two peers in same party simultaneously run `/baloot host`.
- **File:** `Net.lua:816-840`
- **Suggested fix sketch:** When receiving a second `MSG_HOST` from a
  different sender during the same passive phase, surface a dropdown
  in the lobby UI ("Multiple hosts detected: A's game / B's game")
  rather than silently overwriting. Or: refuse a new pendingHost if
  one was already received within the last 10 seconds and warn the
  user.

## AFK handling

### MP-30: AFK auto-play picks lowest TrickRank; not always "polite"
- **Severity:** MEDIUM
- **Symptom:** AFK auto-play picks `bestRank == math.huge` minimum
  TrickRank (`Net.lua:4163-4167`). For a leader who has the only Ace
  of the led suit, this dumps the Ace into a trick they could have
  saved. Not a bug per se — but a "bot would have played differently"
  surprise for the AFK'd human's teammate.
- **Root cause:** Auto-play uses pure-rank minimum, no shape/strategy
  awareness. By design (per the comment at `Net.lua:4001-4002`).
- **Repro:** Human AFKs on trick 1 leading; their hand has an Ace and
  3-4 low cards in the led suit. Auto-play picks the 7 (correct) but
  in cases where they're following a trump-cut they'd lose any
  high card their bot-shaped strategy would have held back.
- **File:** `Net.lua:4153-4198`
- **Suggested fix sketch:** Delegate AFK auto-play to `Bot.PickPlay`
  if `B.Bot` is loaded — same heuristic the seat would have gotten
  if it had been a bot all along. Nontrivial because Bot expects
  hostHands and seats[seat].isBot semantics; would need a
  "synthetic-bot-call" wrapper.

### MP-31: AFK auto-pass on bid is correct, but no "are you still there?" feedback
- **Severity:** LOW
- **Symptom:** Bid timer fires after 60s with no surface chat message.
  Player who came back-from-AFK sees their bid was passed without
  understanding why. The local pre-warn at T-50s does a Sound + Pulse
  (`Net.lua:4033-4036`) but no chat line.
- **Root cause:** `_HostTurnTimeout` logs to L.Info but doesn't print
  to chat (`Net.lua:4147`).
- **Repro:** Open Baloot in lobby, get to your bid turn, alt-tab for
  60+ seconds, come back. Find your bid was passed silently.
- **File:** `Net.lua:4147`
- **Suggested fix sketch:** Print a chat line "<name> passed (AFK)"
  on auto-pass, mirroring the existing Kawesh announcement pattern at
  `Net.lua:3990-3991`.

### MP-32: AFK auto-play on trick-1 also auto-declares melds — handled
- Validated at `Net.lua:4179-4187` per the 50-agent playtest fix.
  Good.

### MP-33: AFK timer is paused-aware but doesn't reset on bot-replace
- **Severity:** LOW
- **Symptom:** A timer was armed for a human seat. The host runs
  `/baloot bots` to fill the seat with a bot (currently this doesn't
  happen mid-game, but future MP-01 fix would). Timer keeps ticking
  on the (now-bot) seat. Eventually fires, finds the seat is now a
  bot, and... the early-return at `Net.lua:4127-4128` doesn't gate
  the FIRE path, only the ARM path. So the timer body runs against
  a bot seat.
- **Root cause:** `_HostTurnTimeout` doesn't re-check `info.isBot`
  before deciding to auto-act. In the BID branch it'd write
  `S.ApplyBid(seat, K.BID_PASS)` regardless of who's now in that
  seat; in the PLAY branch it'd auto-play LOWEST_RANK from
  `hostHands[seat]`, possibly stealing from a bot's strategic plan.
- **Repro:** Hypothetical post-MP-01-fix: human drops, replaced with
  bot, but the AFK timer that was armed pre-drop still fires after
  60s.
- **File:** `Net.lua:4134-4198`
- **Suggested fix sketch:** Add `local info = S.s.seats[seat]; if
  not info or info.isBot then return end` at the top of
  `_HostTurnTimeout`. Same for `_HostBelTimeout`.

## Seat replacement (bot↔human)

### MP-40: No way to swap a bot for a human mid-game (only at lobby)
- **Severity:** MEDIUM
- **Symptom:** Lobby is the only place where seats can be filled.
  Once `/baloot start` runs, `HostHandleJoin` rejects with
  `s.phase ~= K.PHASE_LOBBY` (`State.lua:684`). A friend who logged
  on after the round started can't take over a bot's seat unless the
  game ends and a new lobby is hosted.
- **Root cause:** Single-state machine; PHASE_LOBBY is the only
  seat-mutation gate.
- **Repro:** Game in progress with 1 human + 3 bots. Friend logs
  in. Friend has no way to "take" a bot's seat.
- **File:** `State.lua:683-698`
- **Suggested fix sketch:** Allow late-join during PHASE_PLAY for an
  empty seat AND for a bot seat: open mid-game seat replacement.
  Reset `Bot._partnerStyle[seat]` and `Bot._memory[seat]` at
  takeover; persist the bot's hand (`hostHands[seat]`) and whisper
  the new human their `S.SendHand`. Treat as a host-authorized
  resync.

### MP-41: Lobby-time HostKickSeat removes a human cleanly; no host UI to invoke it
- **Severity:** MEDIUM
- **Symptom:** `S.HostKickSeat` exists (`State.lua:701-704`) but no
  UI button calls it. Host has no way to manually kick a problematic
  human player short of leaving the WoW party. Slash command also
  absent — no `/baloot kick <seat>`.
- **Root cause:** Feature gap. The CHANGELOG documents that
  `K.MSG_KICK = "K"` was REMOVED in v0.11.5 (XR-14) as "kick UX was
  never implemented in the user surface". Confirmed: no UI, no slash.
- **Repro:** Host realizes seat 3 is a troll. Has no in-addon way to
  remove them.
- **File:** `State.lua:701-704`, `Slash.lua` (no kick command),
  `UI.lua` (no kick button)
- **Suggested fix sketch:** Add lobby-only kick button (next to the
  swap button) on each seat, host-only. Or accept the design choice
  to remove the feature; document it as "to remove a player, kick
  them from the WoW party".

## Name / realm display consistency

### MP-50: HostHandleJoin uses raw equality on names
- **Severity:** HIGH
- **Symptom:** `S.HostHandleJoin` (line 685) compares `name == s.localName`
  raw. If `s.localName = "Foo-Realm"` (normalized) but the JOIN sender
  arrives as just "Foo" (same-realm, some WoW client states), the
  guard against self-joining returns falsy → host could end up
  seating themselves AGAIN (already seated at seat 1; HostHandleJoin
  would fail at the next loop because `info.name == name` ALSO uses
  raw equality, so "Foo-Realm" ≠ "Foo" → seat-2 gets assigned to
  "Foo" → now host has TWO seats with their own (different-suffixed)
  name, breaking authorizeSeat.
- **Root cause:** Missing `S.NormalizeName()` calls. The fix is
  applied throughout the receivers (`fromSelf`, `fromHost`,
  `authorizeSeat`, `_OnLobby` peerVersions key, `_OnResyncReq`'s
  `nameEq` helper) but `S.HostHandleJoin` was missed.
- **Repro:** Same-realm friend joins the host's lobby; under certain
  client realm-suffix stripping behavior, the join lands twice (once
  bare, once suffixed). Hard to reproduce without a specific client
  version, but the missing normalization is a real gap.
- **File:** `State.lua:683-698`
- **Suggested fix sketch:** Wrap both equality checks with
  `S.NormalizeName(...)` (cf. the `nameEq` helper at
  `Net.lua:3878-3883`). Same fix for `S.SeatOf` is already applied at
  `State.lua:627-645`.

### MP-51: shortName strips realm; same-realm cross-realm collision in 5-player parties
- **Severity:** LOW
- **Symptom:** Two cross-realm players named "Foo" (one on Realm-A,
  one on Realm-B) join the same party. UI's `shortName` (`UI.lua:389-392`)
  strips realm. Both display as "Foo" in seat labels. Telltale: only
  the partyPanel sidebar uses `m.full` (the original "Foo-Realm"
  form) for `peerVersions` lookup; the seat texts and other UI
  displays use `shortName`. Confusing for the host trying to swap
  the right Foo into the right team.
- **Root cause:** Display gap. `shortName` is used widely in seat
  labels, status messages, and chat prints; only the lobby
  party-panel sidebar disambiguates by full name.
- **Repro:** Two cross-realm "Foo"s in the same WoW party.
- **File:** `UI.lua:389-392` (defn), `UI.lua:2493-2516`, etc. (uses)
- **Suggested fix sketch:** When duplicate `shortName`s detected
  among `S.s.seats`, render with a realm tag suffix
  (`"Foo-RealmA"` / `"Foo-RealmB"`) for disambiguation only at the
  collision sites.

### MP-52: peerVersions normalize-on-write but read-by-full inconsistency
- **Severity:** LOW
- **Symptom:** Line 2916 of UI.lua reads `S.s.peerVersions[m.full]`
  where `m.full = (n .. "-" .. realm)` constructed from `UnitName(u)`.
  But the write side (Net.lua:823, 846, 894-897) uses
  `(S.NormalizeName and S.NormalizeName(sender)) or sender`. If
  `S.NormalizeName` is somehow nil, the key falls back to raw sender
  ("Foo-WowServer-NA" with the WoW server suffix?) which mismatches
  the UI's `(n .. "-" .. realm)` form. In practice `S.NormalizeName`
  is always loaded so this is theoretical, but the asymmetry is
  noted at `Net.lua:891-893`.
- **Root cause:** Conventional but slightly fragile key derivation.
- **Repro:** Theoretical; load-order or partial-install scenario.
- **File:** `UI.lua:2916`, `Net.lua:823-823`
- **Suggested fix sketch:** Use the same `S.NormalizeName(m.full)`
  on the read side too. Symmetric.

### MP-53: GROUP_ROSTER_UPDATE drop detection uses short-name match
- **Severity:** MEDIUM
- **Symptom:** `WHEREDNGN.lua:384-388` uses `info.name:match("^([^%-]+)")`
  to strip realm, then compares against `UnitName(u)` (which itself
  may or may not include realm). When two same-named cross-realm
  players are in the party, a drop of one might NOT trigger a kick
  of the other — fine — but a drop of one might falsely "find" the
  other as still present, masking the real drop.
- **Root cause:** Equality on `short` instead of full
  realm-qualified name.
- **Repro:** Two "Foo"s (different realms) in party; one drops.
  Other "Foo" still UnitExists → loop sees a Foo present → no kick.
- **File:** `WHEREDNGN.lua:384-388`
- **Suggested fix sketch:** Read `select(2, UnitName(u))` for realm
  and compare full normalized form against the seat's normalized
  name.

## Kick / leave / version-mismatch

### MP-60: Version mismatch is cosmetic only
- **Severity:** HIGH
- **Symptom:** UI surfaces version mismatch in red (`UI.lua:2924`) but
  `_OnJoin` doesn't gate on it (`Net.lua:842-861`). A v0.5.0 client
  joining a v1.6.1 lobby silently succeeds. Wire format compatibility
  is "best-effort": pre-v0.7.0 clients lack `MSG_OVERCALL_*` entirely;
  they'll desync into the overcall window, see no resolution, and
  effectively forfeit decisions. CHANGELOG documents the v0.10.2 / v0.10.3
  cross-version "?"-tag dance, but only for one major-version skew.
- **Root cause:** No explicit version check on join. The protocol is
  designed for back-compat (additive fields) but doesn't reject
  truly-old clients.
- **Repro:** Player on old version (e.g., still has v0.10.x cached
  from CurseForge) joins host on v1.6.1; the lobby UI flags the
  mismatch but the join is honored.
- **File:** `Net.lua:842-861`
- **Suggested fix sketch:** Define a `MIN_PROTOCOL_VERSION` in
  `Constants.lua`. In `_OnJoin`, reject joiners whose version is
  below this floor and surface "<name> rejected: outdated addon
  v0.X.Y < required v1.5.0". Whisper a `MSG_VERSION_REJECT` so the
  rejected client also gets a clear surface message.

### MP-61: No /baloot leave command for non-host gracefully exiting mid-game
- **Severity:** MEDIUM
- **Symptom:** A non-host human wants to leave the table (real-life
  interruption, family emergency). They have to either (a) /baloot
  reset locally (which silently breaks the table for the other 3 —
  no notification, the host's view shows them still seated), or (b)
  WoW-leave the party (works because of the GROUP_ROSTER_UPDATE
  handler, BUT triggers the MP-01 dead-seat issue).
- **Root cause:** No `/baloot leave` command. No
  `MSG_PEER_LEAVE` wire message. The graceful-exit-and-replace-with-bot
  flow doesn't exist.
- **Repro:** Player on seat 3 needs to leave mid-round.
- **File:** `Slash.lua` (no leave), `Net.lua` (no MSG)
- **Suggested fix sketch:** Add `/baloot leave`. On non-host: send a
  `MSG_LEAVE_REQ` to the host, who handles by replacing the seat
  with a bot (per MP-01 fix) and broadcasting fresh lobby + a chat
  line. On host: print "as host you can't leave; /baloot reset to
  end the game".

### MP-71: Sticky lobby on remote after host /baloot reset
- **Severity:** MEDIUM
- **Symptom:** Host runs `/baloot reset`. Remote players' UI still
  shows the lobby with seat 1 = host and the gameID, because
  `MSG_LOBBY` was never re-broadcast with a "lobby-gone" signal.
  They have to /baloot reset locally to clear it.
- **Root cause:** Reset is a local operation only; no termination
  wire (cf. MP-22).
- **Repro:** Host /baloot reset in lobby phase.
- **File:** `Slash.lua:115-135`
- **Suggested fix sketch:** Same as MP-22 — emit a `MSG_GAMEEND`
  with an "abort" flag.

### MP-72: Two WoW clients on same machine — both as host of same party
- **Severity:** LOW
- **Symptom:** User runs WoW twice on same machine, logs into two
  characters in the same party. Each runs `/baloot host`. Both
  generate independent gameIDs, broadcast announcements; the
  shared-party context means each sees the other's MSG_HOST as a
  legitimate "remote host". State diverges silently. The
  `fromSelf` guard (Net.lua:782-785) compares against
  `S.s.localName` per-client, and each client has a different
  localName — so neither dedupes the other.
- **Root cause:** Architectural — WHEREDNGN doesn't track that two
  characters belong to the same human. Same as MP-70 but with
  potentially confusing UI on the human (they see two host
  prompts, one from "themselves").
- **Repro:** Run two WoW clients on same machine with two different
  characters in the same party; both run /baloot host.
- **File:** `Net.lua:782-785`, `Net.lua:816-840`
- **Suggested fix sketch:** Same as MP-70 — multi-host detection
  and disambiguation in lobby UI.

### MP-73: Lobby ticker only broadcasts during PHASE_LOBBY; mid-game late-arrivals invisible
- **Severity:** LOW
- **Symptom:** Player joins WoW party after the host already ran
  `/baloot start`. Host stops broadcasting MSG_HOST (the ticker
  cancels at line 73 / line 803-804 when phase != LOBBY). Late-
  arrival has no idea a game is in progress unless they /reload (which
  fires resync via WHEREDNGNDB.lastGameID — but only if THEY have a
  prior session ID matching, which they don't, since they weren't
  here before).
- **Root cause:** Lobby ticker is intentionally lobby-only.
- **Repro:** Friend logs in mid-round of an existing 4-bot game; runs
  `/baloot` to open the addon. Sees an empty lobby, has no way to
  observe the table.
- **File:** `Slash.lua:69-75`, `UI.lua:800-806`
- **Suggested fix sketch:** Spectator mode: if friend tunes in mid-
  game, host can periodically broadcast a low-frequency
  MSG_HOST_ALIVE that surfaces on receivers as "active game in
  progress, see /baloot" — or have the host whisper the snapshot to
  any newly-detected party member who's running the addon. Combine
  with MP-21 host-heartbeat work for one-stone-two-birds.
