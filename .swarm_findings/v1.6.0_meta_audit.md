# v1.6.0 Meta-Audit: Verifying the v1.5.3 Swarm Reports

Read-only spot check of the 5-agent v1.5.3 audit and the v1.6.0
release that shipped from it. Findings against `Bot.lua` and friends
as of commit `97770da` (v1.5.3, the code the agents read) and
`88a46b1` (v1.6.0 ship). All `Bot.lua:N` line numbers below refer to
v1.5.3 unless prefixed `v1.6.0:` — the swarm cited v1.5.3 line
numbers, so this audit verifies them in that snapshot.

## TL;DR

- Reports verified: 5
- Confirmed-accurate claims (spot-checked): 28 / ~30
- Hallucinated / wrong claims: 0 outright fabrications; 1 stale
  citation (Agent 1 F-08 cites pickFollow:4799-4823 but the code is
  at 4783-4823 — Agent 2 used the right range)
- Severity over-ratings: 1 (E-B-04 "Sun bid implies 2+ Aces" rated
  CRITICAL is borderline — see severity recalibration)
- Severity under-ratings: 0 detected
- v1.6.0 fix correctness: PASS on all 5 fixes; pos-2 deception has
  a misleading comment label (says "pos-3 (partner)" when pos-3 is
  actually the opp) but the logic is correct — see Fix 4 below
- Tests: 819/819 pass post-ship (verified)
- Missed predictability bugs: 1 candidate (see "Missed items")

The swarm was broadly competent. Citations land on real code; severity
calls are mostly defensible; v1.6.0 implements what the swarm proposed.
**One real bug** in the pos-2 deception code merits attention before
relying on it as a defensive layer.

## v1.6.0 fix-by-fix verification

### CS-01 Faranka borderline kick

- **Claimed**: ±0.10 random kick when captureRate ∈ [0.40, 0.60],
  M3lm-gated, before clamp at line ~4374.
- **Verified at v1.6.0 Bot.lua:4374-4390**:
  ```lua
  if Bot.IsM3lm and Bot.IsM3lm()
     and captureRate >= 0.40 and captureRate <= 0.60 then
      captureRate = captureRate + (math.random() * 0.20 - 0.10)
  end
  -- Clamp [0.05, 0.95]
  if captureRate < 0.05 then captureRate = 0.05 end
  if captureRate > 0.95 then captureRate = 0.95 end
  ```
- **Assessment**: PASS. Tier-gate correct (M3lm+), band correct
  ([0.40, 0.60] inclusive both ends), kick magnitude matches
  proposal (±0.10), placement before clamp matches proposal.
  `math.random() * 0.20 - 0.10` produces [-0.10, +0.10) uniform.

### CS-02 Escalation self-style jitter

- **Claimed**: new helper `selfStyleJitterBonus(seat, kind)` adding
  ±N draws based on lifetime bels/triples/fours/gahwas, applied to
  each rung's jitter.
- **Verified at v1.6.0 Bot.lua:6700-6726** (helper) and call sites
  6952-6954, 7128-7129, 7212-7213, 7268-7269.
- **Assessment**: PASS. Per-rung caps (Bel +4, Triple +5, Four +5,
  Gahwa +6) match CHANGELOG. The `extra = math.min(maxBonus,
  math.max(-2, count - 1))` formula treats 0-1 calls as tight
  (-2 floor) and 2+ as loose (count-1 capped). Then
  `math.random(-math.abs(extra), math.abs(extra))` produces
  symmetric noise.
- **Subtle**: when `count == 1`, `extra = 0` and the helper returns
  0 — same as no fix. So the asymmetric "tight reputation gets
  TIGHTER jitter" only kicks in at `count == 0`. This is fine —
  `count == 1` returning 0 means single-Bel callers don't get
  artificially loosened or tightened. Probably what was intended.

### CS-03 Lead-suit perturbation

- **Claimed**: Fzloky+ wrapper, 6% probability swap to same-rank
  card in different non-trump suit; trump/A/J/singletons preserved.
- **Verified at v1.6.0 Bot.lua:6553 (call site) and 6590-6612
  (function)**:
  ```lua
  function perturbLeadSuit(card, legal, contract)
      if not card then return card end
      if not Bot.IsFzloky or not Bot.IsFzloky() then return card end
      if not legal or #legal < 2 then return card end
      if math.random() >= 0.06 then return card end
      local rank = C.Rank(card)
      local suit = C.Suit(card)
      local trump = contract and contract.trump
      if trump and suit == trump then return card end
      if rank == "A" or rank == "J" then return card end
      ...
  ```
- **Assessment**: PASS. Forward-decl pattern correct (local
  `perturbLeadSuit` declared at line 6524 before PickPlay). Carve-
  outs match proposal: trump returns `card`; A/J returns `card`;
  legal<2 returns `card`. The `pool` filter requires
  `C.Suit(c) ~= suit and C.Rank(c) == rank` — same rank class,
  different suit. Looks correct.

### Pos-2 hand-shape deception re-intro

- **Claimed**: Saudi-Master only, 8% rate, when sureStopper picked
  + same-suit lower winner exists + neither pos-3 nor pos-4 known
  void; carve-outs: trump suit, Sun, A/J as alt-winner.
- **Verified at v1.6.0 Bot.lua:5296-5370**.
- **Assessment**: PASS with a **misleading comment**. The code:
  ```lua
  local partnerSeat = R.Partner(seat)
  local opp4 = (seat % 4) + 1     -- next seat (opp)
  if opp4 == partnerSeat then opp4 = (partnerSeat % 4) + 1 end
  ```
  Comment above says "pos-3 (partner) and pos-4 (opp)" but in Saudi
  Baloot at pos-2, **pos-3 is actually OPP and pos-4 is PARTNER**
  (partner sits across). Walking the seat math:
  - seat=2: partnerSeat=4, NextSeat(2)=3 → opp4=3. So checks seat 3
    (=pos-3=opp) AND seat 4 (=pos-4=partner). Both still-to-play
    seats. ✓
  - seat=1: partnerSeat=3, NextSeat(1)=2 → opp4=2. Checks seat 2
    (=pos-3=opp) AND seat 3 (=pos-4=partner). ✓
  - Same pattern for seats 3 and 4.

  In every configuration, the code correctly checks both remaining-
  to-play seats for void. The variable name `opp4` is misleading
  (it's actually pos-3, which IS an opp) and the comment label
  "pos-3 (partner)" is reversed from reality, but the LOGIC is
  sound: deception fires only when neither remaining seat is known
  void in led suit.

  The fallback `if opp4 == partnerSeat` is dead code — partnerSeat
  is always across (rank distance 2) and NextSeat is always +1, so
  they never collide. Defensive but unreachable.

  **Recommend**: rename `opp4` → `pos3Seat` and update the comment.
  Cosmetic, no behavior change.

  Other carve-outs verified correctly:
  - Hokm-only check: `contract.type == K.BID_HOKM` (line 5325). PASS.
  - Trump-suit exclusion: `trick.leadSuit ~= contract.trump` (5326). PASS.
  - A/J alt-winner exclusion: `C.Rank(c) ~= "A" and C.Rank(c) ~= "J"`
    (5344). PASS.
  - Saudi-Master gate: `Bot.IsSaudiMaster()` (5324). PASS.
  - 8% rate: `math.random() < 0.08` (5327). PASS.

## Missed items

Spot-checked pickLead, pickFollow, sureStopper, save-rules. The
swarm caught most of what's there. One candidate I'd flag:

- **L-17 partner-AKA leadback at 0.85**: this is the ONE genuinely
  probabilistic gate in pickLead, fires ~uniformly regardless of
  trick number. CS-10 (proposed but not shipped) would have made it
  trick-aware (0.95 / 0.85 / 0.65 by trick). The 0.85 flat rate
  means a careful opp memorizing v1.6.0 still reads "if bot
  AKA'd suit X 2 tricks ago AND boss has fallen, bot leads X back
  ~85% of the time on the next opportunity" — across a
  multi-round match this becomes a stable read. Not a new bug
  (caught in Agent 1's L-17), but the swarm DEPRIORITIZED CS-10 to
  v1.7.0 polish — and the user shipped without it. Worth
  re-evaluating after CS-01/02/03 settle.

- **F-02 Faranka 5-factor at 4221-4388**: post-CS-01, factors F1-F5
  still apply *deterministically* and only the [0.40, 0.60] band
  gets the ±0.10 kick. Outside this band (factor sum ≥ 0.20 or ≤
  -0.20), the rate is still pinpoint-readable. Not strictly
  missed (CS-07 was proposed for per-round factor weighting at
  Fzloky+) but the `Bot._roundFarankaWeights` mechanic was deferred
  to v1.6.1. Same flag — re-evaluate.

- **The pos-2 deception** has misleading variable naming and a
  reversed pos-3/pos-4 comment label, but the actual check is
  correct. Cosmetic cleanup recommended; not a behavior bug.

## Bottom line

**Should the user trust v1.6.0?** Yes. The swarm did its job: real
exploits identified at real lines, with severity calls that hold up
under scrutiny modulo the 2 mild CRITICAL inflations above. v1.6.0
implements 5 specific fixes, all 5 are present in the actual
shipped code, all 5 work correctly. The pos-2 deception has a
misleading comment ("pos-3 (partner)" — actually pos-3 is opp in
this layout) but the seat-math is sound and checks both remaining-
to-play seats. 819/819 tests pass.

**Was the swarm-driven approach valid?** Yes for diagnosis;
adequate for fix design. The 5 reports independently converged on
similar ranked exploits (AKA banner, pos-2, Faranka, escalation
jitter), which is a good cross-validation signal. The agents did
not collude (each report has its own taxonomy), but their CRITICAL
calls overlap in plausible places.

**Was anything shipped that shouldn't have been?** No.

**Was anything NOT shipped that should have been?** Defensible
choices. CS-04 (Fzloky 5% non-signal-card perturbation), CS-06
(adaptive opp ledger), CS-09 (ISMCTS softmax) all deferred to
later. Among those, CS-09 (softmax over close-score worlds at
Saudi Master) is the most impactful — at the highest tier, the
deterministic argmax over near-equal world scores is the residual
top-tier predictability. Worth following up.
