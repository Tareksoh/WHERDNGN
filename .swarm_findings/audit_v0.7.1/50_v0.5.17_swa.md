## 1. R.IsValidSWA pre-existing bug — VERIFIED

`Rules.lua:357-370` (V14 fix) handled `#plays == 4` via
`R.CurrentTrickWinner` BEFORE the empty-hand short-circuit, but the
short-circuit was still un-gated. Pre-fix `if #(hands[caller]) == 0
then return true` fired whenever caller's hand was empty mid-trick
(after caller played their last card as the 1st/2nd/3rd play). The
remaining 1-3 plays of the same trick — including potential opponent
ruff or partner over-take — were never validated. False-positive
SWA. v0.5.17 (`Rules.lua:384`) gates on `#plays == 0` so the
short-circuit only fires BETWEEN tricks. Caught by tests O.2 and O.3.

## 2. Partner adversarial — VERIFIED at Rules.lua:460-466

```
for _, card in ipairs(legal) do
    local nh, ns = applyMove(card)
    if not R.IsValidSWA(callerSeat, nh, contract, ns) then
        return false
    end
end

```

The cooperative branch is gone entirely. The single loop iterates
every legal play of `nextSeat` regardless of team — partner is
treated identically to opponent (any legal partner play that breaks
caller's claim returns false). Symmetric with the prior opponent
branch. Verified diff at `Rules.lua:437-466`.

## 4. Display fix — Net.lua, NOT a UI banner refinement

The "display fix" routes ALL SWA flows through the 5s permission
window (was: `≤3` instant, `≥4` permission). `N.LocalSWA`
(`Net.lua:2092-2117`) drops the `handCount >= 4` clause; condition
becomes `if needPerm then`. `N.MaybeRunBot` bot path
(`Net.lua:3580-3623`) removes the instant-claim branch entirely —
all bot SWAs set `swaRequest` and arm `C_Timer.After`, with the
C_Timer-unavailable test fallback preserved. Reason: the UI banner
only renders when `swaRequest` is non-nil, so instant-claim hid the
caller's cards. Bots auto-accept for ≤3 still works because the
opponent-bot acceptance fires inside the timer window.
