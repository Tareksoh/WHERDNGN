## 2. Scoring discrepancy — what was wrong

Two Net.lua paths used `math.floor((raw + 4) / 10)` (5 rounds DOWN) while `R.ScoreRound` had been corrected to `(raw + 5) / 10` (5 rounds UP per video #43) in **v0.5.6** (commit ac87be4, Rules.lua div10 helper). The two stragglers:

- `Net.HostResolveTakweesh` (Qaid penalty path) — both team-add lines used +4.
- `Net.HostResolveSWA` invalid-SWA branch — both team-add lines used +4.

Effect: a Qaid-penalty resolution or a failed-SWA resolution would round a half-point DIFFERENTLY than a normal round-end with the same raw totals (e.g. raw 75 → 7 via +4 path, → 8 via +5 path). User-reported "scoring not matching the docs."

So the answer to question 2: **both** `Net.HostResolveSWA` (invalid branch) AND `Net.HostResolveTakweesh` had the wrong divisor rounding. The divisor itself (10) was right; the +4/+5 increment was the bug.

## 3. Did these always use +5? — NO, fixed in v0.5.21

Grepped current `Net.lua`:
```

2152: math.floor((rawB + 5) / 10)
2855: math.floor((rawA + 5) / 10)   -- HostResolveSWA invalid branch
```

The diff explicitly converts `+ 4` → `+ 5` in both spots. Pre-v0.5.21 these used +4 (the original pre-v0.5.6 formula); v0.5.21 brought them in line with `R.ScoreRound`. The earlier audit's claim that both "use `(raw + 5) / 10`, matching `R.ScoreRound`" is correct **for HEAD/v0.7.2** but they were NOT always +5 — that's exactly what v0.5.21 fixed.

Audit line numbers (2139-2140, 2843-2844) drifted from HEAD (2151-2152, 2855-2856) due to inserts since v0.5.21; same code.

No stale `(x + 4) / 10` remains in active code (Rules.lua:827 has `+ 4` in a comment-only historical reference; div10 itself is `+ 5`).
