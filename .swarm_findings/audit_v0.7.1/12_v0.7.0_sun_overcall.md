# v0.7.0 Sun-overcall feature audit

Files audited: `State.lua` (884-960), `Rules.lua` (500-563), `Bot.lua`
(2997-3027), `Net.lua` (394-408, 510-520, 1028-1234, 3185-3194, 3403,
1500-1510), `UI.lua` (1318-1365, 1818-1873, 3112-3123), tests
sections P (22) + H (43).

## State machine integrity

Clean. `S.BeginOvercall` refuses non-Hokm / forced contracts; sets
`s.overcall = {decisions={}, bidCard, dealerSeat, startedAt}` and
flips phase. `S.RecordOvercallDecision` enforces lock-out
(line 923) and validates seat/decision. `S.FinalizeOvercall`
correctly mutates contract on UPGRADE (type→Sun, trump→nil, bidder
preserved) and TAKE (also rewrites bidder + recomputes
`belPending`). Phase exits to `PHASE_DOUBLE`, `s.overcall = nil`.

## Bot AI decision logic

`Bot.PickOvercall` is M3lm-gated (D3 spec), respects
`R.CanOvercall`, uses `sunStrength(hand)` against `BOT_OVERCALL_SELF_TH=75`
(bidder UPGRADE) and `BOT_OVERCALL_TAKE_TH=80` (non-bidder TAKE).
Threshold ordering (TAKE > SELF) correct — non-bidder must clear a
stricter bar to seize the contract. Returns "WAIVE" on missing
`hostHands` / non-M3lm tier. No defects.

## Wire protocol invariants

OPEN / DECISION / RESOLVE messages defined in `Constants.lua` 216-225.
Authority: `_OnOvercallDecision` calls `authorizeSeat(seat, sender)`
on the host path; remote echoes gated on `fromHost(sender)`.
Resync replay (`SendResyncRes` 399-408) correctly whispers
`MSG_OVERCALL_OPEN` + recorded decisions to rejoiners.

## UI consistency

`PHASE_OVERCALL` action panel (UI.lua 1818-1873) gates on
`R.CanOvercall(localSeat, contract, oc.bidCard)` for the strong
button. Bidder + Ace bid → only WAIVE shown. Lock-out path (already
decided) renders status text instead of buttons. Host's loopback
skip is patched by explicit `B.UI.Refresh()` in
`_HostBeginOvercallWindow` (1152). Banner self-ticks at 3 Hz with
phase exit hide. `R.CanBel` invariant unaffected — it's checked
in PHASE_DOUBLE which executes after FinalizeOvercall.

## Race conditions / wire-desync

**1 (HIGH).** `_OnOvercallResolve` (Net.lua 1096-1105) **ignores
the wire payload** (`takenStr`, `by`, `otype`) and re-runs
`S.FinalizeOvercall` against the remote's locally-recorded
decisions. If MSG_OVERCALL_DECISION frames are reordered/dropped
relative to MSG_OVERCALL_RESOLVE (chat throttling), the remote
computes a different contract from the host. Self-correcting only
when host's result is `taken=true` (followup MSG_CONTRACT
overwrites). For `taken=false` the host issues no MSG_CONTRACT —
so a remote with an extra/missing locally-recorded decision could
end up at a divergent contract. Defensive fix: have
`_OnOvercallResolve` apply the wire result authoritatively rather
than re-resolving.

**2 (MED).** `S.ApplyContract` idempotence guard (State.lua 971-976)
fires on the post-overcall MSG_CONTRACT broadcast since
`_OnOvercallResolve` already mutated bidder/type/trump to the same
values. Result: SND_CONTRACT cue + `StartLocalWarn("bel")` (line
1004-1006) are skipped. Polish issue, not correctness.

**3 (MED).** `StartLocalWarn("overcall")` arms a `warnAt = 60-10 = 50s`
timer for a 5s window. Window closes long before warn fires, but
the timer is never cancelled and will ping audio + UI 50s into
the next phase. Consider adding `local windowSec` branch in
StartLocalWarn (overcall < 10s, no warn).

## Missed test scenarios

- No headless wire test for the resolve-payload-vs-local-decisions
  divergence (race #1) — feature comments admit "Headless wire
  test absent."
- No test that two simultaneous human TAKE clicks (one bidder, one
  non-bidder) resolve correctly (rejected via order-priority) — the
  H tests only cover deterministic decision sequences.
- No test for resync arriving mid-overcall with partial decisions
  (the 399-408 replay path has no harness coverage).
- No test for the all-bots synchronous-resolve early-close path
  in `_HostBeginOvercallWindow` (line 1138-1141).

## Overall verdict

**Phase 1/2/3 logic is sound.** The state primitives, bot AI, and
the bidder/non-bidder eligibility math are well-tested (65 unit
assertions, exhaustive). One real wire-desync risk exists
(finding #1 above): the remote's authority pathway re-derives
the contract instead of trusting the host's wire payload. Low
real-world frequency but echoes the v0.5.11 Race-A class of bug
(remote-recomputed state diverging from host). Recommend a Phase
4 patch making `_OnOvercallResolve` payload-authoritative + the
StartLocalWarn overcall-timeout fix.
