# v3.0.0 Regression Hunter Audit

Skeptic-mode pass over Bot/Net/UI/State/Constants/Slash/WHEREDNGN/MinimapIcon.lua looking for **regressions introduced by the 12-release marathon (v1.7.0 → v3.0.0, ~93 audit-driven fixes)**. Test harness was 819/819 at every ship; the goal here is to find what the harness can't see.

## TL;DR

- **Total regressions found**: 14
- **CRITICAL**: 1 / **HIGH**: 3 / **MEDIUM**: 5 / **LOW**: 5
- **Top 3 most-actionable**:
  1. **REG-01 (CRITICAL)** — Settings panel uses deprecated `InterfaceOptionsCheckButtonTemplate` (WHEREDNGN.lua:209) — may error on retail 11.x clients on first Esc → Options click.
  2. **REG-02 (HIGH)** — `bindConfirm` doesn't restore button width on click-confirm (UI.lua:1916-1918), only on auto-disarm — pooled buttons remain widened across phases.
  3. **REG-04 (HIGH)** — "New Game" at game-end calls `S.Reset()` host-side without an MP-71-style host-gone broadcast (UI.lua:2632) — non-host peers wait 45s for heartbeat timeout before they can leave.

## WoW API misuse

### REG-01: `InterfaceOptionsCheckButtonTemplate` deprecated/removed in WoW 11.x
- **Severity**: CRITICAL
- **Symptom**: First time a player clicks Esc → Options → AddOns → "Loot & Baloot" on retail 11.0+, addon errors with "Could not find template 'InterfaceOptionsCheckButtonTemplate'" and panel doesn't render. Game itself keeps running.
- **Root cause**: `WHEREDNGN.lua:208-209` — `CreateFrame("CheckButton", nil, panel, "InterfaceOptionsCheckButtonTemplate")` inside `makeCheck`, called 9 times to build the panel. Template was removed when Blizzard moved to the new Settings API (10.0/11.0). The TOC declares `Interface: 120005` (12.0.5 / Midnight) — the deprecated template likely doesn't resolve in this client.
- **From-fix**: v3.0.0 (PJ-04 follow-up + PJ-52). v2.1.0's PJ-04 stub didn't use the template — the v3.0.0 expansion to "real Settings panel with checkboxes" introduced this.
- **Recommended fix**: Switch to `SettingsCheckBoxControlTemplate` for retail or hand-build the checkbox + label as a plain `CheckButton` with a manual FontString (the legacy template's main contribution is the `cb.Text` accessor — easy to replicate). Alternatively, gate on `Settings.RegisterCanvasLayoutCategory` and only build the templated path on legacy clients.

### REG-02: `bindConfirm` doesn't restore button width on click-confirm
- **Severity**: HIGH
- **Symptom**: After confirming any long-label confirm action ("Confirm Bel x3 (closed)?", "Confirm Deny — caller's invalid claim costs them ~30 pts"), the pooled action button remains widened (up to 220px). Next phase's action that reuses the same pool slot — typically a 90px Pass / Hokm / Ashkal button — renders with the leftover width, leaving big gaps and overlapping the next button anchored to its right.
- **Root cause**: `UI.lua:1915-1918` — `if btn.armed then disarm(); fire() end`. `disarm()` resets text and clears timer but never calls `btn:SetWidth(90)`. The auto-disarm timer body at line 1937 DOES call it, so the bug only fires when the user actually completes the confirm. `clearActions()` (UI.lua:1943) and `addAction()` reuse path (UI.lua:1980+) also never reset width.
- **From-fix**: v2.0.0 UX-04. The width-fitting branch was added but reset-on-confirm was missed.
- **Recommended fix**: Add `btn:SetWidth(90)` in `disarm()` (one place fixes both confirm and auto-disarm paths), plus a defensive `b:SetWidth(90)` in `clearActions()` for pool reuse.

### REG-03: `thinkBox` OnUpdate never explicitly cleared (low-cost forever-loop)
- **Severity**: LOW
- **Symptom**: Each seat-badge that has shown a "thinking…" indicator carries a permanent OnUpdate cycling sin() every frame for the addon's lifetime. WoW pauses OnUpdate on hidden frames, so the cost is bounded — but the script is never removed even after the bot turn ends.
- **Root cause**: `UI.lua:3069-3084` — `thinkBox` is created once per seat-badge and gets a permanent OnUpdate. Hide cycle (line 3088-3090) just `:Hide()`s it. Compare to BALOOT's `SetScript("OnUpdate", nil)` after settle (UI.lua:2587).
- **From-fix**: v2.0.1 (BF-10 hotfix wrapping the v1.8.0 FontString crash).
- **Recommended fix**: When transitioning out of bot-thinking, set `b.thinkBox:SetScript("OnUpdate", nil)` to cleanly drop. Optional polish; not a leak in practice since `Hide()` pauses script invocation.

## Frame/timer leaks

### REG-10: Lobby kick is unenforced — kicked client can re-Join immediately
- **Severity**: MEDIUM
- **Symptom**: Host clicks the new "✕" kick button on seat 2 (MP-41 / v3.0). Kicked player's UI shows their seat empty on next MSG_LOBBY. They click "Join" again → `MSG_JOIN` → `_OnJoin` → `S.HostHandleJoin` → re-seated in the first empty seat (the same one they were just kicked from). No host-side ban list.
- **Root cause**: `Net.lua:906-925` (_OnJoin) and `State.lua:683-718` (HostHandleJoin) have no concept of a kicked-name set. `S.HostKickSeat` (`State.lua:722-725`) only nils the seat record.
- **From-fix**: v3.0.0 MP-41. The new Kick button's UI works but the wire-side ban is missing.
- **Recommended fix**: Maintain a per-host `S.s.kickedNames` set populated by `HostKickSeat`; `HostHandleJoin` rejects join if normalized name is in the set. Cleared on lobby reset / new game.

### REG-11: `S.HostKickSeat` doesn't gate on PHASE_LOBBY
- **Severity**: LOW (latent)
- **Symptom**: A future caller invoking `S.HostKickSeat` mid-round would silently brick the seat without any defensive guard. Currently no such caller exists; both UI (`UI.lua:919`) and roster handler (`WHEREDNGN.lua:606`) gate phase externally.
- **Root cause**: `State.lua:722-725` — only checks `s.isHost` and `seat ~= 1`. No phase check.
- **From-fix**: Pre-v1.7.0, but the new MP-41 kick button (v3.0) makes this surface easier to misuse.
- **Recommended fix**: Add `if s.phase ~= K.PHASE_LOBBY then return end` defensively.

### REG-12: `_versionWarnedFor` lives on `S.s` but never reset
- **Severity**: LOW
- **Symptom**: Once a version mismatch is announced, no further warns for the rest of the session even if a new (different-version) host appears. Player switching between hosts only sees one warn per /reload.
- **Root cause**: `Net.lua:861-862` — `S.s._versionWarnedFor = version` set unconditionally; never cleared in `S.Reset` (State.lua:31-139). `/reload` clears it via the fresh `s` table.
- **From-fix**: v2.0.0 MP-60.
- **Recommended fix**: Either reset in `reset()` body, or key by `(host, version)` so different hosts each get their own warn.

## Wire-protocol breakage / version-skew

### REG-20: No regressions found in wire shape
- **Severity**: N/A
- All v1.7+ wire additions (MSG_HEARTBEAT `~` v1.8.0, MSG_LOBBY empty-seats v2.1.0 MP-71) are **forward-compatible**: older clients drop the new tag silently in the dispatch chain (Net.lua:725 falls through to no-op for unknown tags). MSG_LOBBY shape was unchanged — empty-seats is encoded as four empty name fields, decoded by `S.ApplyLobby` as nil seats (which already happens for legitimate empty-lobby states).
- TAKE_HOKM removal (v1.5.3) predates the v1.7+ window — not a marathon regression.
- v3.0.0 ships **no wire-format changes** at all (only UI / settings / kick-button work).

## State-machine edges

### REG-30: BF-15 BALOOT defer + cross-round `s.beloteAnnounced` carryover
- **Severity**: LOW (sound suppression only — scoring path independent)
- **Symptom**: Player who announced BALOOT in round N and again holds K+Q-of-trump in a later round gets no SND_BALOOT cue on the second announcement. The +20 Belote bonus IS still scored correctly (R.ScoreRound counts it via the pre-existing `beloteAnnounced[seat]` truthy check).
- **Root cause**: `State.lua:67` initializes `s.beloteAnnounced = {}` only in `reset()`. `S.ApplyStart` (State.lua:832-930) clears nearly every per-round transient but NOT this map. Across rounds, an announced seat stays announced; the `if S.s.beloteAnnounced[seat] then return end` guard at `Net.lua:236` and the BF-15 deferred guard at `Net.lua:275-276` short-circuit the second announcement.
- **From-fix**: Pre-existing latent bug; v2.3.0 BF-15 deferred-1s timer **didn't introduce this** but uses the same guard so the latent bug surfaces more visibly.
- **Recommended fix**: Add `s.beloteAnnounced = {}` to `S.ApplyStart` body alongside the other per-round clears. Independent fix from BF-15.

### REG-31: MP-04 mid-SWA-permission drop synthesis doesn't trigger resolve
- **Severity**: LOW
- **Symptom**: When a human drops mid-SWA-permission window, MP-04 synthesizes ACCEPT for them (`responses[seat] = true`) but never re-runs the `accepts >= 2` resolve check. The 5s auto-approve timer (already running, set in `_OnSWAReq` at Net.lua:3526) eventually fires and resolves — but the dropped seat's synthesized accept doesn't accelerate the resolution. UX delay up to 5s.
- **Root cause**: `WHEREDNGN.lua:672-680` — assigns directly to `swaRequest.responses[seat]` without going through `N._OnSWAResp` (Net.lua:3532-3610) which is the canonical path that runs the accept-tally + `HostResolveSWA` call.
- **From-fix**: v2.0.0 MP-04.
- **Recommended fix**: Call `N._OnSWAResp("__host__", seat, true, swaRequest.caller)` instead of direct assignment — it routes through the correct gate and handles the accept-count check.

### REG-32: "New Game" host-side reset doesn't broadcast host-gone
- **Severity**: HIGH (UX-bricks remote peers)
- **Symptom**: Host hits "New Game" at game-end. Their state wipes locally (S.Reset → IDLE). Other 3 clients still see PHASE_GAME_END with frozen banner, no Join button (no MSG_HOST), no quick recovery path. They wait up to 45s for the heartbeat timeout (MP-21) to fire `IsHostLikelyGone`, then have to /baloot reset themselves.
- **Root cause**: `UI.lua:2632` — direct `S.Reset()` call, no `N.SendLobby({}, gameID)` broadcast. Compare to `Slash.lua:243-249` (`/baloot reset`) which DOES emit the courteous host-gone signal added in MP-71 (v2.1.0).
- **From-fix**: Inconsistency between v2.1.0 MP-71 (only `/baloot reset` got the broadcast) and the pre-existing UI "New Game" button.
- **Recommended fix**: Before `S.Reset()`, emit `if B.Net and S.s.gameID then B.Net.SendLobby({}, S.s.gameID) end`. Mirror the Slash.lua pattern.

## Race conditions

### REG-40: MP-33 cancels turn timer but doesn't re-arm for newly-installed bot
- **Severity**: MEDIUM
- **Symptom**: When MP-01 mid-round drop replaces a human seat with a bot AND the dropped seat had the active turn, MP-33 (v2.2.0) cancels the host's AFK timer (`Net.lua:CancelTurnTimer`). MaybeRunBot is then called (line 685). But there's a narrow window between `CancelTurnTimer()` (line 632) and `MaybeRunBot()` (line 685) where if MaybeRunBot's bot dispatch fails (pcall'd at Net.lua:5166-5170), there's no AFK timer guard to recover. The synthesis paths for MP-02/03/04 (lines 645-681) run between cancel and dispatch — if any of those error, dispatch never runs.
- **Root cause**: Sequential ordering in `WHEREDNGN.lua:621-685`: cancel → synthesize phase decision → MaybeRunBot. Phase synthesis isn't pcall'd; if it throws, dispatch is skipped, leaving the now-bot seat with no scheduled action.
- **From-fix**: v2.2.0 MP-33 introduced the cancel; the windowed risk was implicit in MP-01/02/03/04's layering.
- **Recommended fix**: Move `MaybeRunBot()` BEFORE the synthesis branches (or pcall-wrap the synthesis). MaybeRunBot is idempotent; calling it twice is safe.

### REG-41: BF-15 deferred BALOOT survives mid-1s host /reset
- **Severity**: LOW
- **Symptom**: If a host calls `/baloot reset` within the 1-second BF-15 defer window after a bot's K+Q-of-trump play, the deferred timer at `Net.lua:269-279` still fires. Guards check `S.s.isHost` (would be false post-reset, so safe) and `S.s.contract` (also nil post-reset), so the announcement is suppressed. **Currently safe** but brittle — any future caller that resets contract without flipping isHost would let the announcement land in the wrong state.
- **Root cause**: `Net.lua:269-279` — defer doesn't carry a generation counter (compare to `B._redealGen` at Slash.lua:255 which IS used for redeal-timer cancellation).
- **From-fix**: v2.3.0 BF-15.
- **Recommended fix**: Optionally use a per-round generation counter for the defer to be defensible. Low priority — current guards happen to cover all known reset paths.

## Saved-vars / persistence

### REG-50: `WHEREDNGNDB.stats` initialization shares mutable default table
- **Severity**: LOW
- **Symptom**: First-session-after-install only. `ensureDB` at `WHEREDNGN.lua:87-89` copies `DEFAULTS.stats` table reference into `WHEREDNGNDB.stats` if nil. Since DEFAULTS is rebuilt each /reload, mutations within the same session leak into the DEFAULTS table — but bound to that session. After SaveSession serializes WHEREDNGNDB to disk and the next /reload deserializes it, the stats table is fresh. Edge case where it'd matter: writing to stats during ensureDB itself (which doesn't happen).
- **Root cause**: `WHEREDNGN.lua:71-78, 87-89` — table literal in DEFAULTS, shared reference assignment.
- **From-fix**: v2.1.0 PJ-40.
- **Recommended fix**: Replace the assignment with a deep-copy: `WHEREDNGNDB[k] = (type(v) == "table") and {} or v` and rely on the per-key fill below. Or rebuild DEFAULTS inside ensureDB. Cosmetic — not actually broken in observable behavior.

### REG-51: Game-end Stats button doesn't type-guard `WHEREDNGNDB.history`
- **Severity**: LOW (requires hand-edit to crash)
- **Symptom**: A user with hand-edited `WHEREDNGNDB.history = "broken"` (string instead of table) crashes on game-end render at `UI.lua:2650` when `#WHEREDNGNDB.history` is computed against a non-table. State.lua:1992 has the type-guard at write-time but UI read at line 2650 doesn't.
- **Root cause**: `UI.lua:2650` — `if WHEREDNGNDB and WHEREDNGNDB.history and #WHEREDNGNDB.history > 0 then` — the `#` blows up on non-table.
- **From-fix**: v2.1.0 PJ-43 (button add); the read pattern is older but this is the new caller.
- **Recommended fix**: `if WHEREDNGNDB and type(WHEREDNGNDB.history) == "table" and #WHEREDNGNDB.history > 0 then`.

## UI render-loop

### REG-60: SWA banner OnUpdate at 0.33s tick triggers full `U.Refresh()` per second-change
- **Severity**: LOW
- **Symptom**: During a 5-second SWA permission window, the banner's OnUpdate at `UI.lua:1725-1768` calls `U.Refresh()` from `populateCards` indirectly (the body refreshes cards if encodedHand changed). Every full Refresh redraws all action buttons, hand cards, seat badges, etc. The implementation looks careful — `_lastEnc` gates re-decode — but each unique encoded hand triggers a full re-layout.
- **Root cause**: `UI.lua:1764-1767` — `if self._lastEnc ~= req.encodedHand then ... self:populateCards(req.encodedHand) end`. populateCards itself doesn't trigger Refresh, so this is mostly safe. UX-33's `if remain ~= self._lastRemain then ... if U.Refresh then U.Refresh() end end` at line 1624-1629 (overcall banner — same pattern) DOES trigger Refresh — once per second during a 5s window. Five Refreshes per overcall window.
- **From-fix**: v2.1.0 UX-33 (made remain countdown drive Refresh). Pre-fix the banner self-rendered without invoking Refresh; the new behavior is needed for the action-button label to show the remain count.
- **Recommended fix**: Acceptable cost (5 Refreshes per 5s window). Could narrow to update only the affected button instead of full Refresh — optional.

### REG-61: AFK pulse `K.UI_AFK_PULSE_TICKS=8 × 0.18s = 1.44s` fires every turn pre-warn
- **Severity**: LOW
- **Symptom**: AFK pulse runs ~8 setBackdropBorderColor swaps over 1.44s every time the local player has 10s left. Cheap; not actually a regression.
- **From-fix**: v3.0 UX-35 (constants extraction); v2.2.0 UX-62 (force-restore).
- **Recommended fix**: None.

## Idempotence

### REG-70: `S.HostHeartbeat` is correctly idempotent
- **Severity**: N/A — verified clean
- `N.StartHostHeartbeat` checks `if N._heartbeatTk then return end` at `Net.lua:3190`. Repeated calls from `HostStartRound` (Net.lua:2303) are no-ops. Self-stops on `phase == IDLE/LOBBY` or `not IsInGroup()` (Net.lua:3196-3199). Looks correct.

### REG-71: Drop-recovery synthesis fires re-entrant `MaybeRunBot` redundantly
- **Severity**: LOW
- **Symptom**: At `WHEREDNGN.lua:653-655`, the escalation-phase branch calls `MaybeRunBot()`. Then at line 684, the OUTER block ALSO calls `MaybeRunBot()`. So in `PHASE_DOUBLE/TRIPLE/FOUR/GAHWA` drop paths, MaybeRunBot fires twice — once for the escalation pickup, then again at the catch-all. Not broken (idempotent), but wasteful — and any bot-decision that's deferred via C_Timer.After in MaybeRunBot's body could double-schedule.
- **Root cause**: `WHEREDNGN.lua:646-686` — phase-specific branches each have their own `MaybeRunBot()` call, then the catch-all also fires. No `else` chain.
- **From-fix**: v2.0.0 MP-02/03/04 (added the inner branches without removing the outer catch-all).
- **Recommended fix**: Convert to `if/elseif/else` so only one MaybeRunBot fires per drop event.

## Bottom line

**Trust verdict**: **Hold a v3.0.1 hotfix before play.**

REG-01 (CRITICAL — `InterfaceOptionsCheckButtonTemplate`) is the same class of bug as the BF-10 v2.0.1 hotfix (WoW API path the test harness can't reach). The v3.0.0 Settings panel is a brand-new feature that **has never been verified in-game** — all 819 tests pass, but the harness stubs CreateFrame. If the user clicks Esc → Options → AddOns → Loot & Baloot on retail 12.0.5 and the deprecated template doesn't resolve, they get a tooltip-spamming Lua error every render frame.

REG-04 (HIGH — game-end "New Game" doesn't broadcast host-gone) is a regression-of-omission from MP-71 (v2.1.0). The same fix applied to `/baloot reset` was not applied to the more-common host-departure path.

REG-02 (HIGH — confirm-button width leak) is a visible UI papercut that surfaces every time a player confirms an escalation chain (Bel x3 / Four / Gahwa) — i.e., constantly. Not a crash but visually broken.

The other 11 regressions (REG-10..71) are MEDIUM/LOW and can be deferred to v3.1. They cluster around drop-recovery synthesis ordering, persisted-state edge cases, and minor UX paper-cuts.

**Recommended action**: tag a v3.0.1 hotfix containing fixes for REG-01, REG-02, and REG-04 before the next playtest. Three changes, all surgical, all in the v3.0.0-introduced code paths. Defer the rest until playtest signal arrives.
